Skills [3/5]

Reusable instruction sets that teach Claude how to behave for a specific task — loaded on-demand to keep things fast and context-efficient.

What is a Skill?

A SKILL.md file that gives Claude a specific persona, workflow and output format for a defined task

Skills are shared across the team — build once, everyone benefits from consistent outputs

ICN examples: "Write NDA", "Deal Brief", "Summarise Meeting", "GPU Proposal", "Outreach Sequence"

Stored in a Project's system prompt or as a pinned instruction set

How Progressive Loading Works

Skills use three-stage lazy loading — Claude only reads what it needs, when it needs it, avoiding context bloat.

1

Metadata — Discovery

Only the skill name + one-line description is loaded at chat start. Claude knows when to use a skill without reading its full contents.

2

Skill Body — Activation

If your request matches a skill, Claude loads the full SKILL.md — instructions, triggers, and guidance.

3

Referenced Files — Execution

Only if needed, Claude reads sub-files (e.g. FORMS.md, scripts/) on-demand — never upfront.

Example Skill File Structure

pdf-skill/
├── SKILL.md        ← main instructions
├── FORMS.md        ← form-filling guide
├── REFERENCE.md    ← detailed API reference
└── scripts/
    └── fill_form.py ← utility script

Claude reads SKILL.md on activation, FORMS.md and scripts only if that step is reached.

Example SKILL.md

# PDF Processing

Extract text and tables from PDF files,
fill forms, merge documents. Use when
working with PDFs, forms, or document
extraction.

## Quick start

import pdfplumber
with pdfplumber.open("doc.pdf") as pdf:
    text = pdf.pages[0].extract_text()

For advanced form filling, see FORMS.md.

Token benefit: Lazy loading means Claude can support hundreds of skills without loading everything at once — no context bloat, no performance hit.

Token Management

What Are Tokens? [1/3]

Tokens are the unit of measurement every LLM uses — understanding them is the foundation of working efficiently with any AI.

01 What counts as a token?

A token is roughly ¾ of a word, or ~4 characters. The model never reads words — it reads tokens.

"Hello" → 1 token

"Artificial intelligence" → 3 tokens

A 1-page email → ~400–600 tokens

A 20-page PDF → ~8,000–15,000 tokens

An image → ~1,000–2,000 tokens (varies by size)

02 How are tokens counted?

Input tokens: everything you send — prompt, files, conversation history, system instructions

Output tokens: every word Claude generates in response

Both count against your limit — longer responses cost more

03 What fills the context window?

The context window is everything Claude can "see" at once. Every item below competes for that space:

💬 Full conversation historyHigh impact
📎 Attached files & documentsHigh impact
🔌 Connector data (Drive, Attio, Gmail)Medium
🌐 Web search resultsMedium
⚙️ System prompt / project instructionsLow–Medium
🧠 Extended Thinking outputVery high
💡

Key insight: Message #1 costs ~200 tokens. By message #30 in the same chat, that same question costs 50,000+ — because Claude re-reads the entire conversation history every single time.

Token Management

General Efficiency Tips [2/3]

Habits that reduce token consumption across any AI tool — not just Claude.

01 Edit, Don't Follow Up

When Claude misses the mark, click Edit on your original message and regenerate — don't send a follow-up

Every follow-up appends to history; editing replaces it

Over 10 rounds, editing alone cuts token use by 80–90%

02 Start Fresh Every 15–20 Messages

Long conversations become exponentially expensive — at message 30, a simple question costs 50,000+ tokens

Ask Claude to summarise → copy → open a new chat → paste as starter context

03 Be Precise, Not Verbose

Specific prompts get better answers and shorter responses

Don't over-explain background unless Claude genuinely needs it

Specify a format: "3 bullet points" not "explain in detail"

04 Reduce What You Share

Extract the relevant section — don't upload the full document

Trim spreadsheets — delete blank rows, keep only the relevant sheet

Convert PDFs to plain text before attaching

Remove images from docs if only the text is needed

05 Use Connectors, Not Copy-Paste

Connectors fetch only what's needed — copy-pasting a full doc forces Claude to read all of it

Paste a URL directly instead of its full text content

Name the specific file in your prompt when using the Drive connector

06 Keep Context Folders Clean

Archive old files so Claude doesn't pull stale or conflicting data

One active version per document — no V1, V2, Final in the same folder

Add metadata headers so Claude identifies relevance without reading everything

Access & Limits

Understanding Your Usage Limits

Claude tracks three distinct limit types — visible at any time under Settings → Usage.

01 Session Limit

A short-burst cap that resets roughly every 1–2 hours

Prevents runaway usage in a single sitting — not a daily or weekly ceiling

If hit: wait for the session reset or switch to a lower-cost model temporarily

02 Weekly Limit

Your primary usage budget — resets on a fixed day each week (visible in Settings)

Applies across all models combined — Opus burns it fastest

If you hit this regularly, consider buying extra usage or upgrading

03 Extra Usage (Paid Top-Up)

Enable "Extra Usage" in Settings to keep working when weekly limits are hit — billed at token API rates

Set a monthly spend cap — ICN policy: max $50 / month

At API rates, $50 ≈ ~100 messages at a heavy 100k-token workload

If you're regularly spending $50+, the $125/mo plan is far more cost-effective — it gives ~200 extra messages per 5-hour window included

Settings → Usage (what you'll see)

⚖️

When to upgrade vs top-up

Top-up first — if you hit your weekly limit occasionally. Set extra usage to $50 max.
Upgrade to $100/mo — only when you're consistently burning through $50 in extra usage month over month. At that point the flat rate is cheaper and you get significantly more headroom.

AI Responsibility
1/2

Errors Compound — Align Before You Delegate [1/2]

A small misalignment at step 1 becomes an entirely wrong outcome by step 10. The more autonomous the task, the higher the stakes.

Interpretation Drift Over Reasoning Steps

Small misalignment at step 1 → compounding drift → wrong outcome by step 10+

⚠ Why agents amplify errors

Each step the LLM takes builds on the last — a wrong assumption in step 1 propagates through every step that follows

With many users or parallel autonomous processes, errors multiply at scale — fast

Models update, prompts drift, data changes, business rules shift — without checkpoints, you won't know when something broke

✓ ICN Rules of Engagement

Align before you delegate — confirm Claude understood the task before asking it to run autonomously

Checkpoint multi-step work — review outputs at each stage, not just the final result

Never skip review on externally-sent outputs — emails, proposals, legal text must have human sign-off

Treat hallucinations as expected, not exceptional — build in verification, especially for facts and figures

🎯

The goal isn't to distrust AI — it's to build workflows where drift is caught early and errors can't compound undetected. Think of it as unit tests for your AI workflows.

AI Responsibility
2/2

Build Fast — Maintain Deliberately [2/2]

AI has dramatically lowered the barrier to build. That's a superpower — but every script, app and automation you ship is software, and software requires ownership.

01 Why "ship and forget" breaks

Packages update and break APIs

Business requirements and data structures change

Third-party integrations are modified or deprecated

Access credentials expire or are revoked

People leave the company — if they own the script, it dies with their account

⚖️

The easier it is to build, the more important it becomes to decide consciously what gets maintained — and what doesn't.

02 The filter: integrate or discard?

Before shipping any piece of code or automation, ask:

Is this business-critical? If it breaks, does work stop?

Will it run repeatedly? One-off scripts don't need the same rigour as recurring automations

Who owns it long-term? If the answer is "no one", don't ship it into production

Is it documented? Can someone else pick it up cold?

If the answer to most of these is unclear → use Claude for a one-off session output, don't automate it yet.

03 Service account ownership

Johannes and Leo are working with Andrei to introduce a shared service account. The rule will be simple:

Any script, Apps Script, or integration built at ICN must share ownership/full access with the service account

This ensures automations survive when people move on — no single point of failure

Applies to: Google Sheets/Apps Script, standalone scripts, any service with an auth token or API key

More detail on the service account setup to follow once confirmed with Andrei

📂 Drive backend — coming soon

Johannes and Leo are also redesigning the Google Drive file/storage/access structure to create a cleaner, more transparent backend — optimised for LLM integrations and the agentic workloads being planned. More information to follow shortly.
