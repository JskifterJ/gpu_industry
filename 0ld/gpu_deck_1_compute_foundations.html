What changes by workload

Training needs long runs and very high inter-GPU communication. Fine-tuning is smaller but still needs reliable memory and bandwidth. Inference needs low latency, high throughput, and strong serving software.

Takeaway

If your goal is user experience and unit economics, optimize inference first. That is where latency, cost per answer, and reliability directly hit product value.

Why GPUs became central

GPUs started in graphics (many pixels in parallel). AI inference also needs large parallel math, so GPUs fit naturally. The key value is not only raw compute, but also memory bandwidth and software stack maturity.

Simple definition

Bandwidth is the speed of data movement. Latency is delay. Throughput is total output.

LLM inference at a glance
05 / 10
From prompt to first token to full answer
Step-by-step inference flow (from v10)
1
Tokenize inputText is split into token IDs from a vocabulary of about 128,256 entries.
2
Prefill passAll prompt tokens run through the model in parallel. This phase dominates TTFT for long prompts.
3
Decode loopOne token is generated at a time, then appended to context. The model runs all layers again for each new token.
4
KV cache growthPast key/value tensors are stored so decode does not recompute history. v10 framing: roughly 0.3 MB per token for 70B-class with GQA.
5
Return streamTokens stream back continuously. A 50-word response is around 67 token passes through model layers.

Red thread takeaway

Long prompts increase prefill and cache pressure. Long outputs multiply decode passes. That is why latency and cost are mostly a memory+bandwidth story, then a compute story.

MFU, parallelism, and interconnect limits
07 / 09
How to fill GPUs efficiently
MFU spelled out
MFU

MFU = Model FLOPs Utilization. It means: the share of a GPU's theoretical math capacity that is actually doing useful model work. In short: higher MFU = less idle silicon + lower cost per token.

Why MFU drops

MFU falls when bandwidth friction is high: large weight streaming pressure, KV-cache growth, small batches, or slow cross-GPU sync. The GPU waits for data instead of doing math.

A100 era

Primary constraint: memory capacity and bandwidth for larger models. Multi-GPU helped, but context-heavy inference still hit HBM limits quickly.

H100/H200 era

Constraint moved to orchestration quality: batching, scheduler policy, and communication overhead now decide whether hardware gains convert to real throughput.

B200 and beyond

Compute and memory jump again, so bottlenecks shift outward: inter-node network fabric, storage/feed pipeline, and software stack efficiency become dominant.

Constraint shift over time (dominant friction)
Interpretation: memory friction decreases per token with newer HBM systems, but interconnect and then full-platform scheduling/storage become the dominant constraints at cluster scale.

Straight summary

AI products are now core daily tools. Under the hood, inference quality and cost are controlled by a practical triangle: model design, memory and bandwidth constraints, and serving software quality. Teams that understand all three can ship faster, serve more users, and spend less.
