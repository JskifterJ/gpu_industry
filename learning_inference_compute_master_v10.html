Before we begin
What exactly is a language model?
A language model is a very large table of numbers called weights (also called parameters). These numbers encode patterns from trillions of words of human text. When you send a message, your words are converted to numbers, those numbers flow through a deep chain of mathematical operations using the weight tables, and a new number comes out the other end — which gets decoded back into a word. There are no rules, no encyclopedias, no lookup tables deliberately programmed in. Just billions of carefully calibrated numbers that, taken together, have compressed an enormous amount of human knowledge into a mathematical structure.

More layers = more reasoning steps. GPT-2 had 12. Each layer refines the vector by connecting words and looking up facts, building toward the final answer.
WIDTH — 8,192 numbers per token
More numbers = richer description of each token. Width determines nuance the model holds simultaneously — and scales the parameter count by d².
The Embedding Space — What the 8,192 Numbers Actually Mean
Every token gets converted into a vector of 8,192 decimal numbers. You can think of this as a location in a 8,192-dimensional coordinate space — an abstract map of meaning. Just as two cities that are geographically close share similar climates and culture, two tokens that are mathematically close in this space share similar meaning and usage patterns.
2D analogy of what 8,192-dim space "looks" like (simplified)
European countries

code keywords
France↔Paris: close

Countries cluster near each other. Capitals cluster near each other. And crucially, the direction from “France” to “Paris” is roughly the same as from “Germany” to “Berlin” — the model learned the relationship “country→capital” as a consistent geometric direction in this space.
Why 8,192 dimensions specifically?
Human language contains an enormous number of distinct concepts, relationships, styles, and logical structures. Researchers found empirically that more dimensions = richer representations = better performance, up to a point of diminishing returns. For a 70B parameter model, 8,192 dimensions gives the model enough “room” to represent thousands of concepts simultaneously without them colliding. Smaller models (7B) use 4,096 dimensions. Larger models may use 12,288 or more.
Distributed representation — no single number means “France”
No individual dimension means “France” or “animal.” Instead, meaning is a pattern across all 8,192 numbers simultaneously — like a chord on a piano. Dimension 42 might contribute a small amount to “France,” and also to “sadness,” and also to “HTML.” This overlapping is called polysemanticity and is why it’s hard to interpret individual neurons.
Nobody wrote these numbers
During training, all 8,192 × 128,256 = ~1 billion embedding numbers started as random values. Every time the model predicted the wrong next token, backpropagation nudged these numbers slightly. After trillions of corrections across trillions of words, the space self-organised: similar tokens naturally migrated to similar locations.

What determines d (hidden dimension)?
Nobody picks d from first principles — it emerges from empirical scaling research. The key insight: larger d lets the model represent more concepts simultaneously without them colliding (the polysemanticity problem). But larger d means the weight matrices grow as d², so each doubling of d costs 4× the memory and compute.
GPT-2 small d = 768
Llama 7B d = 4,096
Llama 70B d = 8,192
GPT-4 est. d ≈ 12,288
Rule of thumb from scaling laws: d should roughly scale as the square root of the total parameter target. Researchers use the Chinchilla optimal training compute budget to decide the right d/L combination.
Why d² creates exponential growth
Every weight matrix connects one d-dimensional vector to another — that's d rows × d columns = d² numbers. Double d, and each matrix has 4× the entries. This is why a 70B model isn't 10× bigger than a 7B model — it's only 2× wider (d: 4,096 → 8,192), but 4× more params per matrix × more layers ≈ 10× total.
Relative weight matrix size (d²) across model sizes:
The d² curve is what makes the difference between a 7B and 70B model so dramatic — not just 10× more neurons, but 10× more connections per layer because each matrix grows in two dimensions simultaneously.
Attention block per layer — 4 × d²
Q matrix (what am I searching for?) d × d
K matrix (what do I contain?) d × d
V matrix (what meaning do I share?) d × d
O matrix (blend all heads together) d × d
Subtotal: 4 × d²  —  at d=8,192: 268M params
FFN (fact-recall) block per layer — ~8 × d²
Expand (d → 3.5d, spread concepts) d × 3.5d
Gate (SwiGLU filter, silences noise) d × 3.5d
Compress (3.5d → d, back to stream) 3.5d × d
Subtotal: ~8 × d²  —  at d=8,192: 537M params
From parameter count to GPU memory
One parameter = one decimal number. To store it you need memory. The format (precision) determines how many bytes each number takes:
70,000,000,000 params
× 2 bytes per param (BF16)
÷ 1,000,000,000
= ~140 GB minimum VRAM
For context: an RTX 4090 gaming GPU has 24 GB. You’d need 6 of them just to load Llama 70B, before serving a single user.
Quantization — trading precision for size
Numbers can be stored at different precisions. FP = Floating Point (scientific notation, flexible range). BF = Brain Float (same range as FP32, fewer fraction digits). INT = Integer (no decimals, fixed range).

What determines the optimal d and L? Scaling law research (Kaplan et al. 2020, Hoffmann et al. 2022 "Chinchilla") shows that for a given compute budget C, the optimal model has roughly d ∝ C⅓ and L ∝ C⅓, meaning width and depth should scale together. Adding more layers with a fixed small d gives diminishing returns; similarly, a very wide but shallow model underperforms a balanced one. In practice, researchers also constrain d to powers of 64 or 128 for GPU tensor-core efficiency.
►Technical detail: where the ~12d² formula comes from
Each Attention block uses 4 weight matrices of size d×d (Q, K, V, O) = 4d². Each FFN block uses expand (d×3.5d) + gate (d×3.5d) + compress (3.5d×d) ≈ 8d². Together: 4 + 8 = ~12d² per layer. Llama 70B: 80 × 12 × 8,192² ≈ 64.4B + ~5.6B embeddings = 70B.

Step 1 — The text gets chopped into numbered chunks
Computers can only process numbers — not letters. Before anything else, your text gets split into small chunks called tokens, and each chunk is swapped for an integer ID from a dictionary. The model never sees your actual words — only these numbers.
Our sentence, split into 5 tokens:
“The”ID 464 “ capital”ID 3361 “ of”ID 286 “ France”ID 4881 “ is”ID 318
Note: the space before “capital” is part of its token — “ capital” and “capital” are different entries. The model works with these exact chunks.
We now have five numbers: [464, 3361, 286, 4881, 318]. But the number 4881 has no inherent meaning — it’s just an index. Next, we swap each number for a rich description of what it means. →
Why not use whole words?
A dictionary of every word in every language would need millions of entries and would still miss typos, technical terms, and code. A 128,256-token vocabulary built from common character combinations covers all major languages, code, math symbols, and rare words (by splitting them into fragments).
On average: 1 token ≈ 0.75 words
A 1,000-word essay is roughly 1,333 tokens. Short common words (“the”, “of”, “is”) are usually 1 token. Long or rare words split into 2–4 tokens. “Tokenization” → [“Token”, “ization”] = 2 tokens.
Step 2 — Each number gets a rich description
The integer 4881 doesn’t tell the model anything about France. In this step, each token ID is swapped for a list of 8,192 decimal numbers — a description of the word’s meaning learned from reading billions of texts. Think of it as GPS coordinates inside an enormous “meaning space”.
Token 4881 (“ France”) becomes a vector of 8,192 numbers:

We now have 5 vectors — one per word. But they’re still isolated islands. “capital” doesn’t know “France” is in the same sentence yet. The next step connects them. →
Nobody wrote these numbers by hand
During training the model read billions of sentences. Words appearing in similar contexts gradually migrated toward each other in this 8,192-dimensional space through a process called backpropagation. After training, “France” and “Germany” are numerically close because they appeared in similar sentences millions of times.
Position also gets encoded here
“Not good” means the opposite of “good not.” Each vector gets a small mathematical adjustment based on its position in the sentence (1st, 2nd, 3rd…) so the model always knows word order, not just word identity.
Step 3 — Words look at each other to understand context
Five isolated descriptions isn’t enough. Meaning lives in relationships. The word “capital” alone could mean the start of a sentence, a financial city, or a type of punishment. Only by looking at “France” in the same sentence does it mean “the seat of French government.” This step builds those connections.

This happens for every word simultaneously. But how exactly does the model decide what’s relevant to what? The next step shows the exact mechanism. →
64 specialists run at once
Llama 70B runs 64 of these connection calculations in parallel — called attention heads. Each head specialises in a different type of relationship: grammar, pronouns, topic continuity, sentiment, cause-and-effect. They all run at the same time and their results are combined.
Runs 80 times
This connection step runs once per layer, 80 times in total. Early layers build grammar connections. Later layers build abstract meaning connections — including recognising that this sentence is asking for a capital city.
The KV Cache grows here
Each new token’s “Key” and “Value” vectors are saved in GPU memory. When generating the next word, the model looks back at all cached vectors instead of reprocessing the whole conversation. This is where the memory constraint in the workload calculator comes from.
Step 4 — The exact mechanism: Query, Key, Value
Every word simultaneously plays three roles. Think of it like a library search: each word is a searcher (Query), a label on a book (Key), and the book’s actual content (Value). Good matches between a Query and a Key mean that book’s content flows into the searcher’s understanding.
The three roles — for “is” asking about “France”:
QUERY — what am I looking for?
“is” asks: I follow a capital-of-country phrase. I’m looking for a city name to complete this.
KEY — what do I contain?
“France” signals: I am a European country. High relevance to capital queries.
VALUE — what meaning do I contribute?
“France”’s full 8,192-number description — all of its geography, culture, and language context — flows into “is”.

We now know how words are connected. But the model still needs to look up the actual answer: which city is France’s capital? That knowledge isn’t stored here. →
All 5 words do this simultaneously
While “is” looks at all others, “France”, “capital”, “of”, and “The” are also simultaneously looking at all other words. Every vector gets updated in the same pass. All in parallel.
The 64-head split
The 8,192-number vector is split 64 ways (128 numbers each). Each head gets its own slice and its own specialisation. Running 64 focused searches of 128 numbers is better than one unfocused search of 8,192 — specialisation beats generalisation here.
Step 5 — Recognising the pattern
Certain heads don’t just connect nearby words — they recognise patterns seen billions of times during training. When our sentence arrives, these heads immediately recognise: this is a capital-of-country question. They’ve seen this exact structure countless times and flag it strongly.
Our sentence triggers a strongly-learned pattern:
“The capital of France is”
The model read “The capital of [country] is [city]” millions of times in training. Certain heads are now wired to fire when they see this exact structure.
These heads signal: pattern match — after “capital of [country] is”, a city name follows. Start looking it up.
The pattern is identified. Now the model needs to retrieve the specific fact: which city? The knowledge is stored in the next component. →
This is why “few-shot prompting” works
If you write 3 examples in your prompt (“Input: X → Output: Y”), certain heads recognise this as a repeated pattern and apply it to new inputs. The model isn’t being retrained — it’s pattern-matching your examples in real time using the same mechanism.
Also works with made-up words
If you write “Glorp translates to Water. Snarv translates to Fire. Glorp translates to ___”, the model predicts “Water” — even though it never saw “Glorp” in training. It’s matching the [A]→[B] pattern structure, not the specific words.
Step 6 — Looking up the actual answer
The attention step built connections between words. But the actual facts — that Paris is the capital of France, that water is H&sub2;O, that Python uses indentation — aren’t stored there. They live in the second component of each layer: the fact-lookup block (FFN).
The “is” vector (now carrying France + capital context) enters the fact-lookup:

“Paris” concept added to the running stream
The “is” vector now encodes: “the answer to this capital-of-France question is Paris.” Steps 3–6 together form one layer. Now we do this 80 times. →
Why expand to 28,672 first?
The 8,192-number vector has many concepts overlapping and tangled together. Expanding to 28,672 spreads them apart, making it possible for a single neuron to encode one specific concept (“French capital”) without it being confused with “German capital” or “French cuisine”.
Nobody programmed these facts
Neuron #14,592 wasn’t assigned to “French capitals.” During training, the model read billions of sentences containing “Paris is the capital of France.” Each time it got this wrong, the weights adjusted slightly. Eventually neuron #14,592 reliably fires for exactly this concept. The model organised its own memory.
~72% of all parameters live here
Most of the 70 billion parameters are in these fact-lookup blocks across 80 layers. The earlier layers store simple facts (“un- means not”). Later layers store complex reasoning (“capital of [country] = its main government city”).
Step 7 — 80 rounds of deepening understanding
Steps 3–6 (connect words + look up facts) form one layer. Llama 70B runs this 80 times. Each pass enriches the vectors further. By layer 80, the “is” vector doesn’t just know it’s a linking verb — it encodes the precise answer the model is about to output.

After layer 80, the vector carries a clear signal: “Paris.” Now we need to turn that signal back into an actual word the user can read. →
Each layer adds, doesn’t replace
Each layer’s output is added to the vector coming in — not replacing it. This is the residual stream. Information from layer 1’s grammar analysis is still present at layer 80. Nothing is discarded.
The hardware consequence
All 80 layers must be streamed from GPU memory for every single token generated. For a 50-word response: ~67 tokens × 140 GB weights = 9.4 terabytes of data moved. How fast data moves from memory to compute is what determines speed — not the raw processing power. You saw this in the prefill/decode section.
Step 8 — Scoring every token in the vocabulary
After layer 80 we have one refined 8,192-number vector. To select the next output, the model compares this vector against every entry in its vocabulary — all 128,256 tokens — and assigns a probability to each. Importantly, the model is not scoring words: it is scoring tokens, which may be whole words, word fragments, punctuation, spaces, or code symbols. The winning token is then looked up in a token→string table to get the actual text to display.
All 128,256 tokens get a probability score. Top token results for our example:

These aren’t just the top 3 — every single token gets a probability. “banana” gets 0.000001%. The model knows Paris is overwhelmingly likely from context.
The winning token “ Paris” is decoded to its string representation via a vocabulary lookup table. Do we always pick the highest-scoring token? Usually — but the next step lets you control this trade-off. →
How the comparison works
The model has an 8,192 × 128,256 table — one column per vocabulary token. The layer-80 vector is multiplied against all 128,256 columns simultaneously, producing a raw score per token. Scores are converted to percentages (all adding to 100%). The column with the highest score = the highest-probability next token.
The output is always a token, not a word
Just like step 1, the output is a token chunk. It might be a whole word (“Paris”), a space+word (“ Paris”), a fragment (“ar”), or punctuation. Your browser assembles the chunks into readable text as they arrive.
Step 9 — How predictable should the answer be?
Everything up to this point is deterministic maths: same input always gives the same probabilities. This final step adds controlled randomness. The temperature dial decides whether the model always picks the most likely word (predictable but potentially robotic) or occasionally picks a less likely one (more varied but riskier).

Top-K=50: keep only the 50 most likely tokens, delete the rest. Prevents truly absurd outputs. Top-P=0.9: keep tokens until their probabilities sum to 90%, then cut. If the model is confident (Paris=92%), only Paris survives. Applied in order after temperature.

THE CORE HARDWARE CONSTRAINT
Every pass loads all 140 GB of weights from HBM. Compute cores are mostly idle, waiting for data to arrive. This memory bottleneck means memory bandwidth, not processing power, determines how fast text is generated.

KV Cache — why it exists
Without caching, pass 67 would need to recompute attention over all 66 prior tokens from scratch — taking 66× longer. The KV cache stores past results in GPU memory so each new token only looks them up. The trade-off: memory consumed is proportional to conversation length.
Batching — the efficiency fix
Instead of one user loading 140 GB of weights at a time, batching runs 32 users’ tokens through the same weight load simultaneously. Same memory bandwidth used — 32× the output. This is why the GPU section shows MFU jumping from 6% to 40-60% with batching. See the workload section for the details.

⚠ bottleneck for 2-node models
For a model spread across 2 nodes (e.g. Llama 405B at ~810 GB), after every one of the 80 layers each node must send ~17 GB of partial results to the other over InfiniBand. At 400 GB/s that takes ~0.04s per layer — 80 layers × 0.04s = ~3.2s per token just in synchronisation overhead if not carefully overlapped. This is why large-scale inference prefers the biggest possible single-node GPUs, and why H100 SXM (with NVLink) commands such a premium over PCIe variants.
Batching
Why serving many users at once is far more efficient
During decode, one user’s token generation loads 140 GB from HBM but only does a tiny matrix multiply. The GPU’s compute cores are idle 94% of the time, waiting for data. Batching runs multiple users through the same weight stream — same memory transfer, far more useful work.
Continuous batching: new requests join the batch mid-generation. As soon as one user finishes, a waiting user takes their slot. GPU is kept busy. Used by vLLM, TensorRT-LLM, all production systems.

Prefill vs Decode — Two Very Different GPU Phases
Click to expand — how prompts are read and tokens generated; why memory bandwidth is the core constraint
Phase 1 — Prefill (reading your prompt)
When your prompt arrives, the GPU reads all tokens simultaneously in one parallel sweep. Processing 10,000 tokens = a 10,000 × 8,192 matrix multiply. The longer your prompt, the heavier this is. This phase is compute-bound.
PROMPT LENGTH vs TTFT
1K tokens
~20–50ms
10K tokens
~200–800ms
100K tokens
5–20 secs
Attention scales with N²: doubling prompt length quadruples the attention work. A 100K-token document = 10 billion relationship comparisons.
FlashAttention (2022) tiles the N×N computation into blocks that fit in fast on-chip SRAM instead of storing the full matrix in HBM. Result: ~8× speedup, no OOM crashes on 100K+ token prompts.
Phase 2 — Decode (generating output)
Once the prompt is read, output is generated one token at a time. Each token loads all 140 GB of weights from HBM to compute cores — but uses only a tiny fraction of the compute capacity. The GPU is mostly idle waiting for data. This phase is memory-bandwidth bound.
MFU vs concurrent users — H100 ×8
1 user: GPU is memory-bound, compute cores mostly idle. 32 users: same 140 GB weight stream serves 32 people → MFU jumps to 40–60%. Batching is the key efficiency lever.
Disaggregation: frontier labs physically separate prefill and decode onto different server clusters. Prefill servers crunch prompts and beam the KV cache over fiber to dedicated decode servers. Decode generation never stalls when a heavy 50K-token prompt arrives.
⚠ The real constraint isn’t having enough memory — it’s how fast you can read it
Once the model fits in GPU memory, the limiting factor is memory bandwidth — how fast the GPU can stream those weights from its memory chips to the compute cores. Doubling the memory doesn’t make it twice as fast. Doubling the bandwidth does.

Forward Pass — What the GPU Actually Does
One complete pass per output token · 80 layers · ~140 GB streamed from memory every time
Every word the model produces requires one full trip through all 80 layers. Think of it like running a very deep assembly line: each station processes your token and passes it on. The slowdown isn’t the work itself — it’s how fast each station can read its instructions from memory.

1.
Your words arrive at the GPU and become numbers
1 user: <1%20 users: <1%50 users: <1%
◆ bottleneck: neither — PCIe is fine for tiny token IDs negligible across all batch sizes
The token IDs (plain integers like 464, 3361, 4881) travel from CPU to GPU memory over the PCIe cable at 32 GB/s. Since integers are tiny, this takes well under a millisecond. Once in GPU memory, each integer is swapped for its 8,192-number description vector from the embedding table. Now the GPU has a numerical representation of your words it can actually process.
2.
80 rounds: connecting the words (Attention layers)
1 user: ~28%20 users: ~33%50 users: ~40%
□ matrix multiply — 64 heads in parallel ◆ memory bandwidth (weights + KV reads grow with batch) rises with batch: KV cache reads scale with users
In each of the 80 layers, the GPU runs the “connect words” step from the walkthrough. It loads four big weight tables (the instruction books for Q, K, V, O) from GPU memory into the compute cores, and the cores do the mathematics simultaneously across 64 specialist channels. The result: each word’s description gets updated to reflect what it learned from all the other words. The bottleneck is loading those tables from memory — the cores are fast, but the memory bus forces them to wait. This is why more memory bandwidth = faster output, not more compute cores.
3.
80 rounds: looking up facts (Fact-recall layers)
1 user: ~58%20 users: ~48%50 users: ~40%
□ matrix multiply — expand then compress ◆ memory bandwidth — largest single cost at low batch falls as batch grows: weight reads shared across users
After connecting words, each layer does a second pass: looking up factual knowledge. The description vector gets expanded to ~28,000 numbers (giving the GPU room to search through many possible facts simultaneously), irrelevant facts are muted, and the result is compressed back down. This is where “Paris is the capital of France” actually gets recalled. This step holds ~72% of all 70 billion parameters — most of the model’s “memory” — and loading those parameters from GPU memory every pass is the single biggest time cost.
4.
GPUs compare notes over the high-speed cable
1 user: ~10%20 users: ~14%50 users: ~16%
◆ bottleneck: neither — NVLink 4.0 is very fast fraction rises slightly as FFN shrinks at higher batch
Because the model is split across 8 GPUs (each holds 1/8 of the weight tables), after every layer each GPU has only a partial answer. They share their partial results over NVLink — a cable running at 900 GB/s inside the server chassis — so every GPU gets the complete picture before moving to the next layer. This happens 80 times per token. Without NVLink (as with the cheaper A10G GPU which relies on the slower PCIe cable at 32 GB/s), this synchronisation step would take 28× longer and cripple performance.
5.
Picking the next word and sending it to your screen
1 user: ~4%20 users: ~5%50 users: ~4%
□ matrix multiply — all 128K tokens scored simultaneously ◆ memory bandwidth (unembedding vocab table) roughly constant across batch sizes
After layer 80, the GPU compares the final description vector against all 128,256 possible words simultaneously, producing a probability for each (“Paris”: 92%, “Lyon”: 7%, everything else: <1%). The temperature setting shapes these probabilities, and the most likely token is selected. The winning token ID is sent back to the CPU over PCIe, decoded back into text, and pushed to your browser — where it appears on screen within milliseconds. Then the whole process starts again for the next word.
Why is generation slow for one user?
Steps 2 and 3 together move ~140 GB of weight data from GPU memory to compute cores — for every single token. The cores sit idle 94% of the time waiting for that data. One user at a time = massive waste. The fix is batching (step 4 above): serve many users through the same weight load simultaneously.
Why does a longer prompt cost more?
During prefill (reading your prompt), step 2’s “connecting words” work scales with N²: 10K tokens = 100M comparisons, 100K tokens = 10 billion comparisons. This is why a 100K-token document can take 5–20 seconds before the first word appears — and why there’s a meaningful cost difference between long and short prompts.

TOKENS PER SECOND (output speed)
H100 node, 70B model, 32 concurrent users: ~420 tok/s total = ~13 tok/s per user. At 0.75 words/token, that’s ~10 words/sec — faster than typical reading speed.
STOP CONDITIONS
Generation stops when: (1) the model produces the end-of-sequence token, (2) max_tokens limit is reached, (3) a stop sequence is matched, or (4) the user cancels the request.

Extended token generation · tool calls · multi-step plans
Models like Claude can generate thousands of “thinking” tokens before producing a final answer. Each thinking token runs the same 80-layer forward pass. A model that thinks for 10,000 tokens before answering is running ~10,000 complete forward passes at full GPU cost.

TOOL CALLS & CONTEXT GROWTH
Every tool result gets appended to the context. A 10-step agent that retrieves 5 documents per step could accumulate 50,000 tokens in the context window. KV cache cost grows proportionally.
WHY REASONING MODELS COST MORE
Thinking tokens are billed at the same per-token rate as output tokens — they require the same GPU compute. A model that thinks for 8,000 tokens then outputs 200 tokens is 40× more compute-intensive than one that just outputs 200 tokens.

Quantization — fitting more into less memory
Model weights are stored as floating-point numbers. The more bits per number, the higher the precision — but also the more memory required. Quantization trades a small amount of accuracy for a large reduction in memory and bandwidth requirements.

FP32 vs BF16 — the standard switch
Training traditionally used FP32 (4 bytes/weight). Modern inference uses BF16 (2 bytes/weight) — half the memory, same dynamic range. The 140 GB figure for Llama 70B is BF16. Virtually no quality loss for inference.
INT8 / INT4 — aggressive compression
INT8: 1 byte/weight = 70 GB. Fits a single 8-GPU node with room for KV cache. Small quality degradation, usually acceptable. INT4: 35 GB for 70B. Fits smaller nodes. Noticeable quality drop on complex tasks — acceptable for many applications.
GGUF / AWQ / GPTQ — quantization formats
Different methods for which weights to quantize and how to compensate. AWQ and GPTQ are activation-aware: they preserve precision on weights the model uses most heavily. GGUF is the llama.cpp community format supporting mixed-precision (some layers INT4, others INT8).

Training vs Inference — How the Model Was Built
Everything above describes inference — running a model that already exists. Training is how the model was created: reading trillions of tokens and slowly adjusting billions of weights until the model predicts text accurately. They share the same forward pass architecture, but training is orders of magnitude more expensive.

TRAINING COST DIFFERENCE
Training requires backpropagation: storing activations from the forward pass (10–20× more memory than inference) to compute gradients. A single Llama 70B training run requires ~10,000 GPU-days. One inference request requires microseconds of GPU time.
FINE-TUNING
A lighter form of training: starting from a pretrained model and running more training steps on a smaller curated dataset. Used to give a model a specific persona, domain knowledge, or instruction-following style. Requires gradient storage but much less compute than full training.
RLHF
Reinforcement Learning from Human Feedback. A reward model scores the base model’s outputs; PPO or GRPO training pushes it toward higher-scoring responses. This is how ChatGPT, Claude, and Gemini learned to be helpful and follow instructions.
