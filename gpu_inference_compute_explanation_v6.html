Step 1 — Tokenization
Before a single GPU wakes up, the CPU splits your text into tokens — integer IDs from a fixed 128,256-entry dictionary. The GPU never sees raw text; only numbers.
"The sky is..."
"The" → int32: 464
" sky" → int32: 6766
" is" → int32: 318
Token IDs are int32 integers — not floats. FP16/INT8 precision applies to weight matrices only.
Tokens are sub-word, not per-word: "hamburger" → ["ham","bur","ger"] (3 tokens). Common short words are usually 1 token.
Uses Byte-Pair Encoding (BPE): an algorithm that learns to merge the most common character pairs until it has a vocabulary of 128,256 chunks — covering all common words across all supported languages.
Average: ~0.75 words per token (or ~4 characters). A 1,000-word essay ≈ 1,330 tokens.
Step 2 — Embedding Lookup
Each integer ID is swapped for a dense vector — 8,192 float16 numbers for Llama 70B. Think of it as GPS coordinates in an 8,192-dimensional meaning space.
ROW 6766 ("sky") — 48 of 8,192 values shown
[0]: +0.342   [1]: -0.118   [2]: +1.204
[3]: -0.067   ...   8,187 more values
THE MEANING SPACE
The "sky" vector sits near "blue," "clouds," and "atmosphere" — far from "apple" or "keyboard." Proximity = learned from context in training: words appearing in similar situations cluster together.
+ POSITION ENCODING
"not" at position 2 means something very different from "not" at position 15. Numbers are added to each embedding to encode its position — so the model knows word order, not just word identity.
Step 3 — Multi-Head Attention: The Reading Team
Instead of one giant attention operation, Llama 70B assigns 64 parallel attention heads. Each head gets a specialised 128-dimension slice of the 8,192-dim vector and learns to look for a different type of relationship.
SLICING: d=8,192 ÷ 64 heads = 128 dims each
H1..H7  H8..H15  H16..H23  H24..H31    ... 64 heads total ...
Head 7 — Grammar Cop
"fired" → links to subject (CEO) and object (manager). Tracks subject-verb-object structure.
Head 23 — Pronoun Detective
"he" → looks back at all prior nouns to resolve the ambiguous reference.
Head 41 — Sentiment Reader
"fired" + "stealing" → routes negative emotional signal forward.
Head 58 — Induction Head
[A]→[B] pattern matching. "Glorp translates to Water" → next time "Glorp" appears, predicts "Water" follows.
WHY MULTIPLE HEADS?
If there were only one giant attention head, all the grammatical, contextual, and emotional signals would collide into a single calculation — the signals would muddy each other. Multiple heads keep distinct relationship types cleanly separated.
STITCHING BACK TOGETHER
All 64 heads run in parallel. Their 128-dim outputs are concatenated back into 8,192 dims, then multiplied by the Output Projection W_o (8,192 × 8,192) — blending 64 specialised perspectives into one coherent representation.
GQA — GROUPED QUERY ATTENTION
Llama 70B uses only 8 KV head groups (vs 64 query heads). 8 query heads share each KV pair. This dramatically shrinks the KV cache — from ~1.25 GB/1K tokens down to ~0.31 MB/token — making long conversations viable.
Step 4 — Q, K, V: What Attention Actually Does
Sentence: "She fished at the river bank". The word "bank" is ambiguous. Attention resolves it by routing information from relevant tokens.
THE THREE MATRICES PER HEAD
Q
Query — what am I looking for? "bank" Q says: "I need financial OR nature context to know what kind I am."
K
Key — what do I contain? "river" K says: "I'm a nature/water token." High dot-product with bank's Q.
V
Value — what meaning do I pass on? If matched, "river" hands its 128 dimensions of meaning to "bank."
ATTENTION SCORES FOR "bank" (after softmax)
"river"
78%
"fished"
15%
others
7%
THE FORMULA
Attention(Q,K,V) = softmax( QKT / √128 ) · V
Q × KT = dot products (similarity scores). Divide by √d_k=√128=11.3 to keep gradients stable. Softmax → percentages. Multiply by V → weighted blend of meanings.
THE RESULT
The vector for "bank" is updated: it now contains 78% of "river"'s meaning, 15% of "fished"'s. The word "bank" has been transformed from a generic noun into specifically "a river bank where someone is fishing." This happens for every token, every layer.
KV CACHE IMPLICATION
The K and V vectors for every token already seen are cached in GPU HBM. Each new token can then "look back" without recomputing. KV cache memory grows linearly with tokens — the hardware constraint you saw in the memory calculator above.
Step 5 — Induction Heads: In-Context Learning
Induction heads are the reason LLMs can follow new patterns in your prompt — even ones that were never in training data. They run a 2-layer circuit called [A]→[B] pattern matching.
Prompt: "Glorp translates to Water. Snarv translates to Fire.
Glorp translates to ___"
LAYER 1 HEAD — "Previous Token" Setup
When processing "Water", this head looks back at "Glorp" (the token before it) and tags "Water" with a hidden marker: "I came immediately after Glorp."
↓ continues through layers...
LAYER 2 HEAD — Induction Head (Execution)
Now processing the second "Glorp". Its Query says: "I'm Glorp. Who has the marker 'I came after Glorp'?" → finds "Water". Pulls Water's Value vector. Predicts "Water" with high confidence.
WHY THIS IS REMARKABLE
The model never saw "Glorp" during training. No weights encode this word. Yet after just one example in your prompt, it can "learn" the pattern. Induction heads are the mechanism behind few-shot prompting — the entire "learn from examples in the prompt" paradigm.
PRACTICAL IMPLICATIONS
Give the model a few examples in your prompt (few-shot prompting), and induction heads copy the pattern — not just the words. "input: X → output: Y" examples work because induction heads reverse-engineer the transformation and apply it to new inputs.
DISCOVERED BY ANTHROPIC RESEARCHERS
Induction heads were identified through mechanistic interpretability — reverse-engineering the model's weights to understand what specific heads actually do. The bigger a model gets, the more induction heads it develops, explaining why larger models are dramatically better at following prompt patterns.
Step 6 — FFN as Memory: Retrieving "Paris"
After Attention routes context between tokens, the Feed-Forward Network acts as a massive key-value memory store — retrieving facts about the world. It holds ~72% of all model parameters.

WHY EXPAND TO 28,672?
The 8,192-dim hidden vector is densely packed — concepts overlap and tangle. Expanding to 28,672 untangles them. In higher dimensions, the model can isolate one specific concept (French geography + capital) without it bleeding into other concepts like "French cuisine" or "European capitals."
SPARSE ACTIVATION — THE KEY INSIGHT
Out of 28,672 neurons checked, only a tiny fraction fire — maybe 2-5%. The SwiGLU activation function zeros out everything else. This sparse activation is what makes the FFN a selective memory retrieval rather than a brute-force lookup.
HOW ARE MEANINGS LEARNED?
Nobody programs neuron #14,592. During training, the model read billions of sentences like "The capital of France is Paris." Each time, backpropagation nudged weights so that the France + capital activation pattern gradually converged on that specific neuron. By the end, it reliably fires for this exact concept.
Step 7 — 80 Layers: The Assembly Line
Each layer runs Attention + FFN once. The same 8,192-dim vector is progressively transformed through all 80 layers. Understanding doesn't exist at layer 1 — it emerges across all 80 together.
L 1–20
Grammar, parts of speech. "sky" = noun, "is" = linking verb.
↓
L 21–60
Relationships, co-reference. "sky" + "is" → needs a weather/colour descriptor.
↓
L 61–80
Abstract reasoning, intent, style. "is" vector now encodes "current state of a celestial object."
RESIDUAL STREAM
Each layer adds its output to the incoming vector (residual connection). This means information from early layers isn't lost — it flows through to the end. The residual stream accumulates meaning across all 80 layers rather than rewriting from scratch each time.
EARLY vs LATE LAYERS — WHAT'S DIFFERENT?
Early-layer FFNs store simple facts: "un- = negation." Late-layer FFNs store complex reasoning: "France + capital = Paris." Ablating (removing) early layers hurts syntax. Ablating late layers hurts factual retrieval. They are genuinely different in what they learn.
HARDWARE CONSEQUENCE
All 80 layers must be streamed from HBM for every single token generated. That's 140 GB of weights loaded for each of the 270 passes needed to produce a 200-word response. Memory bandwidth — not compute — is the bottleneck.
Step 8 — Logits & Softmax: From Math to Words
Layer 80's output vector (8,192 numbers) is multiplied against the un-embedding matrix (8,192 × 128,256) — one column per vocabulary token. The result: one raw score (logit) per token.

RAW LOGITS BEFORE SOFTMAX
Logits are raw scores — unconstrained numbers. "Paris" might score +25, "Lyon" +18, "banana" -40. Softmax applies exp() to each score and normalises to 1.0 — turning unconstrained numbers into interpretable probabilities.
TOKENS, NOT WORDS
The model selects from 128,256 tokens — not words. The selected token might be a whole word ("Paris"), a suffix ("ly"), punctuation, a space+word combination (" Paris"), or a code symbol ("def"). Average: ~0.75 words per token.

TOP-K: HARD CUTOFF
Top-K = 50: sort all 128,256 tokens by probability, keep the top 50, delete the rest. Renormalise to 100%. Prevents the model going completely off-rails while still allowing creativity within the top candidates.
TOP-P: DYNAMIC CUTOFF
Top-P = 0.9: keep tokens until their cumulative probability reaches 90%, then cut. If the model is confident (Paris = 92%), only 1 token survives. If uncertain, hundreds survive. More adaptive than Top-K.
PIPELINE ORDER
1. Compute raw logits
2. Apply Temperature (divide logits by T)
3. Apply Top-K (cut long tail)
4. Apply Top-P (dynamic nucleus)
5. Roll weighted die → output token

Prefill vs Decode — The Two GPU Phases
MFU · TTFT · FlashAttention · Disaggregation
Phase 1 — Prefill (Reading)
When your prompt arrives, the GPU reads all tokens at once in a single parallel sweep. For a 10,000-token RAG document, it multiplies a 10,000 × 8,192 matrix. This phase is compute-bound — Tensor Cores are working hard.
O(N²) ATTENTION COST
1K tokens
1M ops · ~20ms
10K tokens
100M ops · ~200ms
100K tokens
10B ops · ~20s!
Doubling context length quadruples the attention computation. This is the TTFT bottleneck.
MFU: HIGH (60-80%) — compute cores redlining
Phase 2 — Decode (Writing)
Once prefill is done, the model generates output one token at a time. Each token needs all 140 GB of weights streamed — but barely any math. The GPU is mostly idle, waiting for memory. This phase is memory-bandwidth-bound.
GPU UTILISATION DURING DECODE
~6% compute utilisation per token (memory-bound)
Batching is the fix: serve 32 users per weight load → same memory transfer, 32× the useful output. MFU climbs to ~40-60%.
MFU: LOW (5-8%) single user → 40-60% batched
MFU — Model FLOPs Utilisation

Before 2022: a 100K prompt would crash the GPU (OOM — the N² attention matrix was larger than all of HBM). FlashAttention solved this.
FlashAttention — The Memory Crash Fix
Standard attention tries to compute the full N×N grid of token relationships and store it in HBM simultaneously. At 100K tokens, that matrix is terabytes — larger than all GPU memory.
BEFORE: Standard Attention
Computes full N×N matrix → writes to HBM → reads back → normalises. Huge memory footprint. Crashes on long contexts. Many slow HBM round-trips.
↓ 2022: FlashAttention paper
AFTER: FlashAttention (tiled)
Breaks prompt into small tiles. Loads one tile into SRAM (fastest GPU memory, ~20 MB). Does all the math. Writes only the final result to HBM. Deletes tile. Next tile. Never holds the full N×N matrix.

Prefill–Decode Disaggregation
The problem: mixing prefill and decode on the same server causes stutter. A sudden 50K-token RAG prefill from User B pauses User A's token stream mid-sentence — the heavy matrix math monopolises the GPU.
① Prefill Servers
Dedicated GPUs only crunch incoming prompts. High compute utilisation. Never generate text. Build the KV Cache.
KV Cache beamed over high-speed fiber
② Decode Servers
Only generate tokens. Batch hundreds of users. Never interrupted by prefill math. Smooth, uninterrupted token streams.
Used by Anthropic, Google, and other frontier labs serving at scale. The infrastructure reason responses feel smooth even under heavy load.

WHY A10G FAILS FOR 70B WORKLOADS
Memory: 24 GB GDDR6 means 6 chips just for weights. 8 chips with context. This is already an entire server rack just for one user's request.
Interconnect: No NVLink — all GPU-to-GPU communication goes through PCIe at 32 GB/s. 80 AllReduce operations per token across 8 GPUs via PCIe adds seconds of overhead per response.
Bandwidth: 600 GB/s vs H100's 3.35 TB/s. Each weight load takes 5.6× longer. For a 200-word response: H100 ~11s · A10G ~63s. With PCIe AllReduce overhead: effectively unusable.

132 Streaming Multiprocessors
Each SM contains Tensor Cores — specialised matrix multiply units. All 132 fire simultaneously on different matrix shards. During inference, SMs are often idle waiting for HBM data — the GPU is memory-bound, not compute-bound.
KV Cache (inside HBM)
Grows with context + users
Keys + Values for every processed token. Prevents recomputing attention from scratch. PagedAttention (vLLM) manages it in virtual memory pages. When full: no new requests can start — primary cause of "queue full" errors.
Model Weights (inside HBM)
Resident across all requests
Pre-loaded once. All users share the same weights — no per-user copies. For 70B FP16 across 8 GPUs: 17.5 GB each. Weights never change during inference — only during training.

▸ Weight Loading — Every Forward Pass (bandwidth deep dive)
THE WAREHOUSE ANALOGY
HBM = a warehouse next to the factory. Model weights live in the warehouse permanently — they never leave HBM across requests. But for every forward pass, every matrix multiply requires physically streaming weights from HBM into SM compute registers. The weights do not stay "in" the compute units between tokens.

d = hidden dimension — Llama 70B uses d=8,192. This is the number of "neurons" per layer. Doubling d quadruples parameter count (d² scaling). FFN holds ~72% of layer params; attention holds ~28%.

Quantization — Precision, size and quality trade-offs
What quantization actually does
Quantization rounds every weight to a coarser decimal precision. It does not remove learned patterns or delete knowledge — it reduces the fidelity of every one of the 70 billion dials. Fine distinctions degrade first; core knowledge is largely preserved at INT8.
ONE WEIGHT AT DIFFERENT PRECISIONS
FP320.004317524 bytes — full precision
FP160.004322 bytes — standard inference
INT80.0041 byte — good quality
INT40.000.5 bytes — 4x smaller
Analogy: JPEG image quality. 100% vs 80% vs 20%. Same image — less detail at the edges. INT8 ≈ 80% quality (most users cannot tell). INT4 ≈ 50% (noticeable in edge cases).

What degrades first
INT8 (good quality): Rare edge cases, subtle reasoning chains, complex maths. Most users cannot tell the difference from FP16.
INT4 (moderate quality): Multi-step logical reasoning, precise instruction following, rare vocabulary. Noticeable in benchmarks.
INT2 (aggressive): Core knowledge impacted. Frequent factual errors. Only useful under extreme memory constraints.
The Large Context Problem — RAG, documents and KV cache pressure
Modern enterprise prompts — RAG plus uploaded documents plus MCP plus conversation history — routinely produce 10,000–50,000 token contexts. Each token requires a Key and Value tensor in the KV Cache, stored in GPU HBM for the entire request duration. This stacks on top of the 140 GB of model weights already resident in HBM.

Same 3-chip minimum for memory. But A100 at 2.0 TB/s vs H100 at 3.35 TB/s means each weight load takes 67% longer. Tokens per second roughly halved for the same model.
KV CACHE GROWS LINEARLY
Every new output token adds another entry. A 14K input generating 500 tokens adds ~1.5 GB more HBM during generation. When the KV cache fills: no new requests can start. This is the primary cause of "server at capacity" errors.

Bottlenecks & Where to Spend — Ranked by ROI
Ranked by impact on inference throughput. Addressing bottlenecks from the top gives the best return on hardware spend.
🔴
HBM Bandwidth (within chip)
Upgrade to H200 or MI300X
The dominant bottleneck. Every forward pass streams all 140 GB of weights from HBM to the SM compute units. H100 (3.35 TB/s) to H200 (4.8 TB/s) gives roughly +43% tokens/second for the same model. This is the highest-ROI hardware upgrade for inference.
🔴
HBM Capacity (model + KV cache)
B200/MI300X (192 GB) or quantize
70B FP16 plus 14K context equals 184 GB — 3 H100s per user. Hard ceiling on concurrent users. Solutions: larger GPUs (B200/MI300X = 192 GB each), quantization (INT4 cuts to 35 GB), or smaller models.
🟡
RAG Network Latency
Co-locate vector DB with GPU
If the RAG vector database is on customer infrastructure and GPU inference is at the provider, every request adds 20-200ms round-trip — often exceeding GPU compute time itself. The most overlooked bottleneck in enterprise deployments.
🟡
InfiniBand (inter-server AllReduce)
Fewer, larger GPUs = less IB
Required when a model spans multiple servers (e.g. Llama 405B across 7 nodes). ~50 GB/s effective vs NVLink 900 GB/s. Compounds across 80 layers times 270+ tokens per response.
🟡
Attention O(n^2) with context length
Smaller chunks + Flash Attention
Doubling context from 7K to 14K tokens quadruples attention compute and doubles KV cache memory. RAG pipelines that retrieve large chunks and long histories directly degrade throughput.
🟢
Raw TFLOPS (compute power)
Not worth spending on alone
GPU compute units are idle waiting for HBM data during inference. Adding TFLOPS without adding HBM bandwidth produces near-zero improvement. An A100 (312 TFLOPS) is slower for inference than an H100 (989 TFLOPS) — the gap comes from HBM bandwidth, not compute.
🟢
NVLink (intra-server)
Not a priority
900 GB/s within a single DGX server. AllReduce after each of 80 layers adds under 5% total overhead. Only matters when comparing to A10G (PCIe only) — not a bottleneck in normal H100/A100 configurations.
