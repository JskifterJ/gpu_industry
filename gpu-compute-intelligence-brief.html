01

The Full Stack,
Explained

Understanding why hardware and software are inseparable — and why the car analogy breaks down

The popular mental model of a GPU cloud provider is simple: buy expensive hardware, rent it out, profit. It's the cloud-as-commodity thesis — silicon as a standardized unit of exchange. This is wrong, and expensively so. The providers who understand this distinction are pulling away from those who don't.

The GPU is not a static resource. It is a programmable parallel processor whose effective output — measured in tokens/second, model accuracy, or training throughput — is determined as much by the software layer running on it as by the silicon itself. A misconfigured CUDA kernel can waste 60% of theoretical FLOPS. An unoptimized attention mechanism can cut inference throughput by 4x. A poorly managed KV cache can make a 400GB HBM3e memory pool feel like 40GB.

The stack is seven layers deep, and every layer leaks performance. The relationship between them is not additive but multiplicative — inefficiency at one layer compounds through all layers above it.

Notice that Layers 2–4 — the compute primitives, inference engines, and orchestration — represent the widest gap between theoretical GPU performance and realized business value. This is where the competitive battles of 2025–2027 will be fought. Not in the datacenter, but in the compiler.

First Principles

An H100 SXM5 delivers 3.35 PFLOPS of BF16 tensor compute. A naively-deployed inference stack will utilize ~30–40% of that. A production-grade stack with FlashAttention-3, FP8 quantization, continuous batching, and speculative decoding can push utilization above 70%. That delta is not a hardware decision — it is a software engineering decision, and it has a direct line to dollar-per-token economics.

02

Emerging Workloads:
What Actually Runs

Training is 15% of the story. Inference, agentic pipelines, and synthetic data are the other 85%.

The GPU market in 2022–2023 was defined by the training race. Frontier labs consumed compute like oxygen — every new cluster announcement measured in H100 equivalents. That race hasn't stopped, but it has become a smaller fraction of total GPU demand as the inference economy has exploded underneath it.

Understanding the distinct nature of each emerging workload class is essential — not just commercially, but physically. Each places a different stress on the compute-memory-bandwidth triangle, and therefore requires different hardware configurations, software optimizations, and pricing structures.

The Data Behind "Training is ~15–20% of GPU Demand"

This claim is now well-supported across multiple independent sources. NVIDIA's own CFO commentary (Q3 FY2025) noted that inference has overtaken training as the primary driver of data center GPU deployment, with inference workloads representing "the majority of our shipments going to hyperscalers." Andreessen Horowitz (2024 AI Report) estimated inference accounts for ~80% of enterprise AI compute spend. Goldman Sachs (AI Infrastructure 2024) projected that by 2025, inference would represent 65–80% of total GPU hours consumed in production deployments. SemiAnalysis ClusterMAX 2.0 survey data (84 providers, 140+ customers) shows the largest capacity reservations are now dominated by inference-optimised cluster configurations rather than training configurations. The mechanism is straightforward: a frontier model is trained once (weeks) but then served billions of times (indefinitely). Even a modest inference API serving 10M daily active users will accrue more GPU-hours per month than the original training run within 6–12 months of launch.

The Agentic Inflection

Agentic workloads are architecturally distinct from standard inference. A single user query in an agentic pipeline may invoke a model 8–40 times, accumulate a context window of 100k–1M tokens, and maintain KV cache state across tool calls. This fundamentally changes memory requirements: where a batched inference job is memory-bandwidth-bound (streaming weights), an agentic job is memory-capacity-bound (KV cache growth). The H200's 141GB HBM3e is targeting this exact use case.

The agentic transition also reshapes pricing. Today, GPU cloud is priced per-GPU-hour. But for an agentic pipeline, the relevant unit is the cost-per-task-completion — a compound metric that includes compute, context length, number of inference hops, retrieval costs, and network latency between components. Providers who can offer a coherent per-task pricing model gain a structural commercial advantage, because they can align incentives with outcomes rather than just with silicon hours.

Where Does
Compute Happen?

Hypercloud, sovereign cloud, on-device, and the edge — a four-way fracture that changes everything

The question of where compute runs is not merely technical — it is geopolitical, economic, and architectural simultaneously. The emerging answer is not "in the cloud" but rather "across a tiered hierarchy", with intelligence distributed according to latency requirements, data sovereignty constraints, model capability needs, and cost tolerance.

Hyperscale Cloud
AWS / GCP / Azure · Frontier training, enterprise APIs, highest reliability · 30–40% of GPU revenue today
GPU Cloud Tier-2

Sovereign / National AI

Enterprise On-Prem
NVIDIA DGX SuperPOD, private clusters · Regulated sectors (finance, healthcare, defence) · CAPEX-heavy but total control
Edge / Telco

On-Device / Consumer
Apple Silicon NPU, Qualcomm Hexagon, Samsung NPU · 1B–13B quantized models · Privacy-first use cases

The critical insight is that these tiers are not competing — they are complementary and increasingly composed. A production application in 2026 might route a privacy-sensitive query to an on-device model, a domain-specific query to a fine-tuned enterprise instance, and a complex multi-step reasoning task to a frontier cloud model. The intelligence layer decides where to route, not the user.

This has a profound implication for the GPU cloud market: the competitive moat is no longer "who has the most H100s" but "who has the lowest-friction, most intelligent routing infrastructure across the entire compute hierarchy."

04

Big vs. Small:
The Model Size Question

Scaling laws are not dead — but the frontier is forking. The answer is "both, for different reasons."

The debate has been framed as a binary: will AI models keep growing in size, consuming ever-more-compute, or will efficiency improvements enable "good enough" small models to run on local hardware? This is a false dichotomy. The answer is a deliberate stratification, driven by economic and architectural logic.

The case for scale-up

Scaling laws (Chinchilla, Kaplan) have not been falsified — they've been refined. Post-training techniques (RLHF, DPO, GRPO) dramatically extend the capability surface of a given parameter count. But for tasks requiring deep multi-step reasoning, long-context synthesis, and novel problem-solving, larger models with more parameters still produce qualitatively different outputs. GPT-4 class models didn't emerge from distillation — they emerged from scale. The frontier will continue scaling toward 1T+ parameter MoE architectures with selective expert activation.

The case for small / on-device

DeepSeek's V3 showed that a 671B MoE model trained with aggressive efficiency techniques (FP8 mixed-precision, Multi-head Latent Attention, auxiliary-loss-free load balancing) could match GPT-4-class performance at a fraction of the inference cost. Llama 3.3 70B runs comfortably on a single H100 or two A100s. Quantized to Q4_K_M (llama.cpp), a 7B model runs in real-time on Apple M2 Pro. Privacy requirements, latency constraints, and cost sensitivity will drive enormous demand for small, highly capable, locally-deployable models.

The physical mechanism enabling small model viability is quantization — specifically the shift from FP32 → FP16 → BF16 → FP8 → INT4 in the precision with which model weights are stored. The practical effect is a 4–8x reduction in memory footprint with often less than 2% degradation on standard benchmarks for models above ~7B parameters. The software toolchain — GPTQ, AWQ, GGUF (llama.cpp), ExLlamaV2 — has matured to the point where quantized deployment is no longer expert-only.

The key physical constraint for on-device inference is not compute but memory bandwidth. During autoregressive generation (token-by-token), the GPU/NPU must stream the entire weight matrix from DRAM into compute units on every single forward pass. An Apple M4 Pro with its 273 GB/s unified memory bandwidth can sustain ~50–80 tokens/second on a 7B Q4 model — usable, but not enterprise-grade throughput. The Qualcomm Snapdragon X Elite (LPDDR5X at 135 GB/s) is the floor; workstation-class Apple Mac Studio (M4 Ultra, ~800 GB/s) starts to approach serious performance.

The MoE Unlocking

Mixture-of-Experts architectures are the most important architectural development in large model scaling. By activating only a subset of experts (e.g., 8 out of 64) per token, MoE models achieve dramatically higher parameter capacity (more knowledge, better reasoning) without proportionally higher inference compute. The catch: all parameters must reside in memory, even if only 10–15% are used per forward pass. This means MoE inference is even more aggressively memory-capacity-bound than dense model inference — directly driving demand for H200 and B200 class GPUs with 141GB+ HBM. The DeepSeek R1/V3 architecture is a masterclass in making this work economically.

05

Software Is
Eating the GPU

The inference engine, not the chip, is becoming the primary determinant of unit economics

The conventional hardware story says NVIDIA wins because CUDA is the moat. This is partially true but increasingly incomplete. The real moat is CUDA plus the entire software ecosystem that builds atop it — and that ecosystem is now producing performance gains that rival semiconductor process node improvements. Software is delivering hardware-equivalent gains on annual cycles, while hardware delivers them on 2–3 year cycles.

# Inference throughput evolution on same H100 SXM5 @ constant INT8 quantization

naive_pytorch → ~600 tok/s # Baseline: HF generate(), single request, eager mode
+ torch_compile → ~900 tok/s # torch.compile() fuses ops, reduces kernel launch overhead (~1.5x)
+ continuous_batch → ~3,200 tok/s # vLLM: serve N requests in parallel, interleave prefill/decode (3.5x)
+ paged_attn → ~4,800 tok/s # PagedAttention: near-100% GPU mem util vs ~60% naive (~1.5x)
+ speculative_dec → ~7,200 tok/s # Draft-then-verify: batch verify N tokens per step (~1.5x)
+ radix_cache → ~9,600 tok/s # SGLang: skip recomputing shared prefixes across requests (~1.35x)

# Note: FP8/INT4 quantization changes the weight precision, not a pure software opt.
# These gains are measured at constant INT8 weight quantization across all steps.

TOTAL SW GAIN: ~16x throughput on identical hardware, identical model weights
COST IMPACT: $0.32/1M tokens → $0.02/1M tokens # Same silicon · different stack

This has a direct implication for GPU cloud economics: the provider who owns the optimized inference stack captures margin that the naive hardware renter cannot. Two providers running the same H100 hardware can have a 5–14x difference in token output per dollar, depending entirely on their software layer. This is where vLLM, SGLang, and TensorRT-LLM are not merely engineering choices — they are P&L decisions.

Key Software Innovations · 2023–2025
PagedAttention (vLLM)
── NAIVE APPROACH ────────────────────────────── GPU Mem: [WASTED][Req A ][WASTED][Req B ][WASTED] fragmentation ↑ ~35% wasted ── PAGED ATTENTION ───────────────────────────── KV Cache as virtual pages (like OS virtual memory): [A-pg1][B-pg1][A-pg2][C-pg1][B-pg2][A-pg3][C-pg2] ~98% utilization · no pre-allocation waste
The KV cache stores all past token attention states so the model doesn't recompute them. In naive implementations, each request pre-allocates a contiguous memory block for its maximum possible length — most of which sits empty. PagedAttention stores KV cache in small, non-contiguous "pages" mapped via a page table (exactly like virtual memory in an OS). Pages are only allocated when needed. Result: near-zero fragmentation, enabling 3–4x higher concurrent request capacity on identical VRAM.
Speculative Decoding ↑ 1.5–3x latency
── WHY GENERATION IS SLOW ────────────────────── LLM decoding is sequential — one token at a time: Step 1: model(tokens[0..N]) → token[N+1] # full forward pass Step 2: model(tokens[0..N+1]) → token[N+2] # full forward pass again Step 3: model(tokens[0..N+2]) → token[N+3] # ...every. single. token. ── SPECULATIVE DECODING ──────────────────────── Draft model (tiny, fast) proposes K tokens speculatively: draft → ["The", "sky", "is", "blue", "and"] # 5 guesses Target model (full LLM) verifies ALL K tokens in ONE pass: verify → [✓ "The", ✓ "sky", ✓ "is", ✓ "blue", ✗ "and"] Accept first 4, resample token 5 → net cost: ~1 forward pass for 4 tokens instead of 4 forward passes. ~4x faster.
Autoregressive generation is inherently serial — each token requires a full forward pass through the entire model. Speculative decoding breaks this bottleneck by using a tiny "draft" model (e.g., a 7B model drafting for a 70B target) to propose K tokens in K cheap forward passes, then the large model verifies all K in a single batched forward pass. Since the large model's verify pass is batched, GPU utilization spikes. When the draft model is accurate (high acceptance rate), wall-clock latency approaches the draft model's speed despite using the full-quality target model. The EAGLE-2 variant uses learned draft heads that achieve 3–4x speedup with >90% acceptance rates on chat models.
RadixAttention / Prefix Caching (SGLang) ↑ 1.3–4x for shared prompts
── THE PROBLEM ───────────────────────────────── Request 1: [SYSTEM PROMPT 512 tok][User query A][Answer A] Request 2: [SYSTEM PROMPT 512 tok][User query B][Answer B] Request 3: [SYSTEM PROMPT 512 tok][User query C][Answer C] Without prefix caching: compute system prompt KV cache 3× = 3 × 512 tok of wasted FLOPS ── RADIX ATTENTION ───────────────────────────── KV cache organised as a radix (prefix) tree: root └─[SYSTEM PROMPT]──────────────── cached once ✓ ├─[User query A]─[Answer A] ├─[User query B]─[Answer B] └─[User query C]─[Answer C] Shared prefix computed once, reused across all requests. For 100-req/s with 1k-token system prompts: saves ~100k tok/s
Every LLM request with a system prompt (instructions, RAG context, tool definitions) must compute the KV cache for that prefix from scratch in naive serving. RadixAttention organises the KV cache as a radix tree where shared prefixes are stored and reused across requests. For production deployments — where hundreds of concurrent requests share the same system prompt — this eliminates the dominant source of redundant compute. In agentic pipelines where the context window carries 10k–100k tokens of shared history, the gains compound dramatically: some benchmarks show 4–6x throughput improvement when prefix reuse rate is high.
FlashAttention 2/3
── STANDARD ATTENTION MEMORY PATTERN ────────── Q, K, V matrices → compute S = QKᵀ → write S to HBM (slow) → read S from HBM → softmax(S) → write to HBM → read from HBM → multiply by V HBM bandwidth bottleneck: O(N²) reads/writes for seq len N ── FLASHATTENTION: SRAM TILING ───────────────── Divide Q,K,V into tiles that fit in SRAM (fast, on-chip) Fuse softmax + matmul into single CUDA kernel per tile: tile₁ → compute partial attention in SRAM → accumulate tile₂ → compute partial attention in SRAM → accumulate ... → final output written to HBM once HBM reads: O(N) instead of O(N²) — critical for long context
Standard attention computes the full N×N attention matrix and materialises it in GPU HBM (slow off-chip memory). For a 32k-token context, this is a 32k×32k matrix — roughly 4GB of intermediate data being written and read repeatedly. FlashAttention tiles the computation so intermediate results stay in fast on-chip SRAM, reducing HBM bandwidth by up to 10x for long sequences. FA3 specifically targets the H100's new async copy units and Tensor Core pipelining. Without FlashAttention, running 128k+ context windows would be economically prohibitive on current hardware.
A Note on Quantization

Quantization (FP32→FP16→BF16→FP8→INT4) is not a software serving optimization — it is a change to the model's weight precision, made at deployment time and applied uniformly. It trades a small accuracy loss for a large reduction in memory footprint and bandwidth pressure. The throughput gains in the terminal block above are measured at constant INT8 quantization to isolate the pure software serving improvements. Quantization is separately powerful — a BF16→INT4 conversion halves memory, allowing twice the batch size on the same hardware — but it belongs in the model preparation layer, not the inference server layer.

The CUDA Lock-in Question

NVIDIA's CUDA moat is real but eroding at the margins. AMD ROCm 6.0 has closed the correctness gap and now runs most major inference frameworks. Intel OneAPI is less mature but strategically important. More critically, OpenAI Triton — a Python-based GPU kernel language — is enabling kernel-level optimizations to be written once and compiled to CUDA, ROCm, and future backends. If Triton reaches critical mass, it structurally weakens NVIDIA's software moat even while its hardware moat remains intact.

The Compiler Layer Battle

MLIR, XLA, TVM, and IREE represent the emerging compiler backends that can target diverse accelerators from a single model representation. Google's XLA has proven this works at scale for TPUs. The question for 2026–2027 is whether open compiler infrastructure matures sufficiently to route workloads intelligently across NVIDIA, AMD, and emerging ASIC accelerators — creating genuine hardware competition for the first time since CUDA's emergence.

Where in the Stack
Are Competitors Playing?

Four archetypes: infrastructure maximalist, sovereign/geographic, orchestration platform, and inference specialist — each with a distinct recent strategic trajectory

Understanding a GPU cloud company requires mapping them precisely on two axes: where in the stack they create and capture value, and what customer segment they're optimizing for. The providers who try to cover all layers without differentiation will be squeezed between hyperscalers (who have endless capital) and specialized players (who have genuine software depth). The winning positioning in 2025–2027 will require a deliberate bet on one or two stack layers, executed with depth.

The strategic imperative for CoreWeave is to move up the stack into inference software before its hardware differentiation compresses. Their CoreWeave Inference product is the right direction; the question is execution speed. A secondary risk is NVIDIA's own cloud ambitions (DGX Cloud), which competes directly with CoreWeave's core offering.

Inference kernel engineering depth — custom CUDA kernels tuned per-model rather than generic frameworks. First-class speculative decoding, continuous batching, and prefix caching implementations. Strategic Analysis
Fireworks represents the most technically interesting competitive bet: own the inference engine layer without owning any hardware. This is the purest software-margin play in the market. The risk is that the inference API becomes commoditised as open-source frameworks (vLLM, SGLang) mature and providers increasingly self-host. Fireworks' defense is kernel-engineering depth that takes years to replicate — but that advantage narrows as Triton and other compiler abstractions lower the barrier to custom kernels. The FireFunction and agentic tooling bet is astute: structured output + tool-call inference is genuinely harder than standard generation and commands premium pricing.

Reading across these four archetypes illuminates the structural logic of the market. CoreWeave is a capital-intensive infrastructure bet that requires hardware scarcity to persist long enough to generate returns before contracts renew. Nebius is a geographic/regulatory arbitrage play with a time-bounded window before hyperscalers close the compliance gap. Modal is a developer platform play requiring developer experience to compound into enterprise lock-in. Fireworks AI is the inference-layer pure-play — highest software margins, most exposed to commoditisation of serving frameworks.

The most defensible long-term position is one that owns a meaningful chunk of the inference engine layer while building adjacent data infrastructure — because that combination sits directly between hardware commoditization (threatening CoreWeave-style plays) and application-layer fragmentation (threatening orchestration plays). The data infrastructure angle is explored in the next section.

Object Storage:
The Invisible Infrastructure Layer

Where our existing storage offering integrates with the GPU compute stack — and why it is a structural complement, not a commodity add-on

Object storage is typically treated as a utility — an S3-compatible endpoint bolted onto a GPU cloud offering as a checkbox feature. This framing misses the actual technical and commercial role that object storage plays in AI workloads. Storage is not peripheral to the GPU stack; it is the data substrate on which every layer above it depends. A GPU sitting idle waiting for data from slow object storage is a GPU generating zero revenue. The latency and throughput characteristics of the storage layer directly set the ceiling on training efficiency and inference serving quality.

Training Data Ingestion
Multi-TB datasets streamed to GPU workers during training. Bottleneck: sustained read throughput. Object storage must sustain >10 GB/s per H100 node or GPUs stall waiting on data.
Model Checkpointing
Training runs checkpoint every N steps. A 70B model checkpoint = ~140GB (BF16). At 8 GPUs/node, checkpointing to slow object storage stalls training for minutes per save. Fast object storage = faster recovery, more frequent safe-points.
Model Weight Serving
Inference servers cold-load model weights from object storage. A 70B model = 140GB; initial load from S3-equivalent: 2–8 min. From local NVMe: 20–40s. Storage proximity to GPU is a direct latency-to-revenue factor.
RAG Vector + Document Store
Retrieval-Augmented Generation pipelines store document chunks and metadata in object storage, retrieved per-query. Low-latency object read is critical for sub-100ms RAG response. Tiered storage (hot/warm/cold) maps naturally here.
Synthetic Data Pipelines
Post-training synthetic data generation produces massive output volumes (100B+ tokens). These must be written to object storage, deduplicated, filtered, and re-ingested. High-throughput write + metadata management are the key requirements.
Inference Result Caching
Semantic caching stores prior LLM outputs keyed by embedding similarity. Object storage + vector index enables cache hits that bypass GPU compute entirely. At scale, 20–40% of queries can be served from cache.
Integration Opportunity: Where We Play

Our existing object storage offering has a natural insertion point at four critical junctures in the GPU compute workflow: (1) Training dataset hosting — co-located storage eliminates egress fees and maximises read throughput to GPU workers; (2) Checkpoint storage — high-throughput write path with versioning and lifecycle policies for automatic tier-down; (3) Model weight registry — a content-addressed model store where weights are versioned, deduped across fine-tune variants, and served via a low-latency presigned URL mechanism; and (4) Inference artifact storage — output logs, evaluation results, and generated data from production inference pipelines. The competitive advantage is not the S3 API (that is table stakes) — it is storage co-location with our GPU infrastructure, which eliminates cross-provider network hops and their associated latency and cost.

The Co-location Moat

When compute and storage live in the same datacenter and are connected by high-speed internal fabric (rather than crossing the public internet), three things change: latency drops from ~50ms to <1ms for metadata operations; throughput rises from ~500 MB/s to >10 GB/s for large sequential reads; egress cost drops to zero (no cross-provider bandwidth charges). For a training cluster reading 10TB of data per training step, this is not a minor optimisation — it is the difference between a GPU utilisation rate of 40% and 85%.

Pricing Lever: Egress-Free Storage

The dominant hyperscaler model charges $0.08–$0.09/GB for data egress — a cost that compounds rapidly for AI workloads that read datasets repeatedly during training. A 10TB dataset read 50 times (one epoch per day for 50 days) generates $40,000–$45,000 in egress fees on AWS S3 alone. Positioning our object storage as egress-free when paired with our GPU compute is a commercially differentiated offer that directly attacks the hyperscaler lock-in mechanism and aligns with how customers experience cost.

The competitive landscape on storage-compute integration is instructive. CoreWeave recently launched CoreWeave Object Storage powered by VAST Data — specifically to address the training data bottleneck for their large-cluster customers. Nebius has Object Storage as a first-class product integrated with their AI platform. Modal added Volumes (persistent, distributed storage) as a foundational feature because they found that GPU compute without integrated storage was insufficient for real workloads. The pattern is consistent: every serious GPU cloud is building tighter storage integration, because the customers demand it.

The strategic question is not whether to offer object storage — it is how deeply to integrate it into the inference and training workflow, and whether to position it as a standalone SKU or as a bundled compute+storage unit that competes on total cost rather than per-service price.

Strategic Thesis:
The Compounding Stack

Where the white space is, what the durable moats look like, and what a well-positioned startup should be building
"The GPU cloud market is not a hardware rental market. It is a software-defined performance arbitrage market, and the companies that understand this first will capture margins that are structurally unavailable to those who don't."
— Editorial Thesis · The Compute Stack Brief, Vol. III

The convergence of trends analyzed in this brief points to a clear strategic map for the next 36 months. Five forces are reshaping value capture across the stack:

1. Hardware will partially commoditize, but software won't. The H100/H200/B200 supply situation will gradually normalize through 2026, driven by AMD MI300X adoption, Nvidia Blackwell ramp, and reduced frontier lab dominance over total demand. As unit economics of raw GPU hours compress, the margin migrates upward — to inference engines, orchestration, and specialized model hosting.

2. Agentic workloads will force a re-architecture of inference infrastructure. The current generation of inference servers (vLLM, TGI) was built for stateless, single-turn requests. Agentic pipelines require stateful, multi-turn, tool-calling, long-context, low-latency sessions. This is a genuinely different engineering problem — one that creates white space for new tooling. The KV cache management problem alone (maintaining state across 50+ tool calls in a single session) is not well-solved by any existing production system.

3. Quantization will make on-device/edge viable for 80% of use cases. By 2026, a sub-$2,000 consumer device will be able to run a Q4-quantized 30B model at conversational speed. This is not a threat to the cloud — frontier model training and complex multi-step reasoning will remain cloud-bound. But it fundamentally changes the competitive surface for commodity API tasks (summarization, classification, simple Q&A), which currently represent the majority of inference API revenue for mid-tier cloud providers.

4. Sovereign AI is a durable structural force, not a temporary phenomenon. The EU AI Act, US export controls, India's data localization push, and Gulf state national AI strategies are all creating demand for compute that is jurisdictionally constrained. This is not going away — it's accelerating. The first well-capitalized cloud provider in each major jurisdiction will lock in long-term government and enterprise contracts that are difficult to displace.

5. The inference engine is the new operating system. Just as in the 1990s, the OS became the platform on which all applications ran and extracted rent accordingly, the inference engine — the software layer that translates a model weight file into production API calls — is becoming the platform on which AI applications are built. vLLM is open-source but companies like Anyscale (its primary commercial backer) are building managed services atop it. TensorRT-LLM is NVIDIA's attempt to own this layer. The company that owns the default production inference engine wins the same way Microsoft Windows won.
