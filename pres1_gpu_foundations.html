Why each component matters
CPU — the general-purpose brain
8–128 powerful cores. Each does one complex task at a time — web browsing, OS, spreadsheets. Too slow for AI: a 70B model needs billions of identical maths ops simultaneously.
RAM — temporary workspace
16–64 GB on a typical PC. Holds the data the CPU is actively working on. Too small and too slow for AI: a single 70B model weighs 140 GB.
GPU — built for parallel maths ★
Originally designed for rendering game pixels — thousands of identical calculations fired at once. AI matrix multiplication is exactly the same problem. The GPU was repurposed and the world changed.
CPU vs GPU — the fundamental difference
CPU — few powerful cores
4–128 cores. Each handles complex sequential logic independently.
GPU — thousands of tiny cores
16,896 cores. All run the same maths simultaneously — perfect for AI matrix operations.
Key takeaway: CPUs coordinate and sequence; GPUs execute huge batches of the same operation. LLM inference is dominated by that second pattern.

Applications across industries
The same chip runs chatbots, drug discovery, and self-driving cars.
GPUs power every compute-intensive AI application. LLM inference is the largest share today, but the same silicon also runs biology, physics, image synthesis, and autonomous systems.

LLM inference dominates GPU-hours and is growing fastest. Image/video gen and scientific computing form a meaningful and rapidly expanding share of the "Other" bucket.
Language & Reasoning
LLM Inference ★ Dominant
Chat assistants, code copilots, document analysis, API calls. Every Claude, ChatGPT and Copilot query runs on GPU clusters. ~65% of all GPU-hours. Memory-bandwidth-bound (see Slide 7).
Agentic / Multi-Step
AI agents that call tools, browse the web, write and execute code — repeating 8–40+ inference calls per user task. Fastest-growing demand segment. 3× YoY growth per Anthropic / OpenAI disclosures.
Model Training
Building a model from scratch. GPT-4 class: thousands of GPUs, months, $50–100M. Compute-bound; requires NVLink/InfiniBand clusters. Declining share as inference proliferates.
Image, Video & Audio
Image Generation
Diffusion models (Stable Diffusion, DALL·E, Midjourney) run hundreds of denoising steps per image — each a full forward pass through a U-Net or DiT. 1 image ≈ 3–8 seconds on an A100. Heavily compute-bound unlike text inference. Stability AI, Black Forest Labs.
Video Generation
Generating a 5-second 720p video requires ~100–1,000× the compute of a single image. OpenAI Sora, Runway Gen-3, Google Veo 2. A single video can use thousands of H100-hours. Compute demand growing ~10× per year. Source: Epoch AI compute trends.
Audio & Speech
Real-time speech-to-text (Whisper), text-to-speech (ElevenLabs, XTTS), music generation (Udio, Suno). Speech models are smaller (~1–3B params) but run at very high concurrency for consumer products — high aggregate GPU demand.
Science & Engineering
Protein Folding & Drug Discovery
AlphaFold 2/3 (DeepMind) predicted the structure of every known protein in hours — a task that would have taken millions of lab-years. GPU-accelerated molecular dynamics enables drug screening at scale: Pfizer, Novartis, Schrödinger. Source: Nature, 2021.
Physics Simulation
Climate modelling, fluid dynamics (CFD), materials science, nuclear fusion (ITER uses GPU clusters for plasma simulation). NVIDIA's Modulus framework runs PDE solvers 1,000× faster than CPU-only solvers. National labs (ORNL, Argonne) dedicate entire clusters to physics.
Autonomous Vehicles & Robotics
Tesla Dojo, Waymo, and robotics firms run petabytes of camera/lidar data through perception models continuously. Training a new AV model requires 10,000–100,000 GPU-hours. NVIDIA DGX clusters are standard infra for every major AV programme. Source: Tesla AI Day 2022.
Key takeaway: these very different products all reduce to the same primitive: large matrix multiplications in parallel. That shared math is why one architecture (GPU) can dominate workloads from protein folding to chat responses.
Sources: SemiAnalysis GPU demand breakdown (2025); Epoch AI "Compute Trends" (2024); Nature "AlphaFold2" Jumper et al. (2021); Tesla AI Day compute estimates (2022); Anthropic / OpenAI API volume disclosures.

Model Size: Layers × Width × Precision = GPU Memory Required
05 / 13
Interactive: build your own model — watch parameters and memory respond
Parameters = Layers × Width² × 12
Memory = Parameters × bytes-per-weight
Key takeaway: width ($d$) is the expensive knob because memory grows with $d^2$, while depth ($L$) grows linearly. Precision decides whether the model fits on one GPU or many.
d — Hidden Dimension
The length of each token's vector. Every layer multiplies a d×d matrix — so doubling d quadruples parameters per layer. Llama 70B: d = 8,192.
L — Layers (Depth)
Each layer runs one full attention + FFN pass over all tokens. More layers = deeper reasoning. Parameters scale linearly with L. Llama 70B: L = 80.
Precision — bytes/weight
FP32 = 4 bytes, BF16 = 2, FP8 = 1, INT4 = 0.5. Same model at FP8 uses half the GPU memory as BF16. Memory = params × bytes.

The AI Server: Rack → 8-GPU Tray → Single Chip
06 / 13
A three-level blowout
From Server Rack to Single Chip: Three Levels of AI Hardware
4 Critical GPU Components
01
HBM — High Bandwidth Memory
The memory bottleneck. 80 GB, 3,350 GB/s. Every token generated must stream all model weights through here. The primary constraint on inference speed.
02
Tensor Cores — Matrix Engines
The compute engine. 528 units. Each performs a 4×4 matrix multiply in one clock cycle — the exact operation at the heart of every transformer layer. 1,979 TFLOPS at FP8.
03
NVLink — Inter-GPU Highway
The inter-GPU link. Connects all 8 GPUs in a node. Without NVLink, sending layer activations between GPUs is cripplingly slow. With it, 8 chips act as one.
04
Node Interconnect - Inter-Node Bottleneck
The scale-out constraint. Inside one node, 8 GPUs are tightly coupled. Across nodes, bandwidth and latency degrade sharply, so larger models on older hardware do not scale linearly by just adding more GPUs.

DXG vs. HGX:
10 / 13
Turnkey system vs OEM reference architecture
DXG vs. HGX:
DGX is NVIDIA turnkey, HGX is OEM-customizable infrastructure.
DGX
NVIDIA turnkey system
Pre-built, ready-to-use system sold directly by NVIDIA. Best fit when teams want fast deployment and consistent performance without hardware integration work.
System price: ~$300k-$400k (8x H100 class)
Price per GPU: ~$37.5k-$50k
GPU-to-GPU: NVLink up to 900 GB/s (effective ~810 GB/s at 8 GPUs)
HGX
OEM reference architecture
Customizable platform for OEMs like Dell, HPE, and Supermicro to build flexible cloud-scale systems. Better for tailored networking, storage, and rack-level design.
System price estimate: ~$300k-$500k (8x H100 HGX, OEM-config dependent)
Price per GPU estimate: ~$37.5k-$62.5k (including integration, varies by OEM)
GPU-to-GPU baseline: PCIe ~64 GB/s (effective ~11.5 GB/s at 8 GPUs without full NVSwitch)
Pricing note: DGX launches at ~$399k; HGX-based systems (Dell, HPE, Supermicro) overlap or exceed DGX pricing due to OEM margins and integration. Advantage: HGX modularity, not cost. Key difference: interconnect bandwidth scales throughput, not price.
Difference in practice
H100 system comparison (DGX H100 vs H100 PCIe) · Llama 70B FP8 · 8x H100 · TP=8 · Time500 = TTFT + 500×decode-step (economics model)
DGX H100 (green): higher NVLink bandwidth keeps utilization high as batch rises, so time to 500 tokens grows slowly.
H100 PCIe (red): weaker interconnect causes larger sync overhead at higher batch, so utilization drops and time to 500 tokens rises faster.
Why HGX efficiency is lower (in one line)
For LLM inference, token generation is synchronization-heavy: HGX over PCIe spends a larger slice of each step on GPU-to-GPU coordination, so Tensor Cores idle more often. DGX over NVLink cuts that sync tax and converts more wall-clock time into actual token throughput.

Decision rule: optimize for cost per delivered token at your real batch/context pattern, not headline $/GPU-hour.
How to maximize MFU (practical playbook): keep model weights a small share of node HBM (roughly 20-35%), then push KV cache usage via batch size and context length until total HBM sits near the safe edge (about 85-95% used). If weights dominate HBM, you are memory-read limited; if total usage is too low, Tensor Cores idle.
