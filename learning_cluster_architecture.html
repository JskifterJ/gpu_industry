NETWORK GAP: Customer RAG ↔ Provider GPU
For enterprise deployments, the RAG vector DB often lives on customer infrastructure while GPU inference runs on the provider's cloud. Every request requires a network round-trip to retrieve RAG chunks — typically 20–200 ms — which commonly exceeds the GPU computation time itself. Co-locating the vector DB with the GPU cluster is the primary fix.

3.35 TB/s bandwidth. Data moves from HBM to the compute units at enormous speed. This bandwidth — not raw processing power — is the primary limit during inference. Upgrading to H200 (4.8 TB/s) is the single highest-ROI hardware move for inference speed.
132 Streaming Multiprocessors
Each SM contains Tensor Cores — specialised matrix multiply units. All 132 SMs fire simultaneously on different parts of the same operation. This is why GPUs are ~100× faster than CPUs for AI. During inference they are often idle waiting for HBM data to arrive — the GPU is memory-bound, not compute-bound.
KV Cache (inside HBM)
Grows with context + users
Stores attention Keys + Values for every processed token so they need not be recomputed on the next forward pass. PagedAttention (vLLM) manages it in memory pages. When full, no new requests can start — this is the primary cause of inference server "queue full" errors.
Model Weights (inside HBM)
Resident across all requests
Pre-loaded once at server start. All user requests share the same weights — nobody gets their own copy. For 70B FP16: 140 GB total spread across 8 GPUs (17.5 GB each). Weights themselves never change during inference.
④ Forward Pass — repeated once per output token
1
Token Embedding Lookup
Each input token ID maps to an 8,192-dimensional vector (70B model). These embedding vectors are loaded from a lookup table in HBM into the SM registers — the raw numerical starting point before any transformer computation begins.
2
Multi-Head Self-Attention ⚠ O(n²)
Every token is compared to every other token in the context. For 14,000 tokens: 196 million pairwise comparisons per attention head, across 64 heads. Each comparison uses Q, K, V weight matrices (8,192 × 8,192 = 67M params each). The attention results (Keys and Values) are written to the KV Cache so they are not recomputed for the next token. This is the most memory-intensive step. Flash Attention 2 reduces peak HBM usage via tiled computation.
3
Feed-Forward Network (FFN)
Two enormous matrix multiplications per transformer layer: expand (8,192 → 28,672) then contract (28,672 → 8,192). These weight matrices hold the bulk of what the model "knows" from training. Executed as cuBLAS GEMM operations across all 132 SMs simultaneously. Most of the 140 GB of weights consumed here. HBM bandwidth is the bottleneck.
4
NCCL AllReduce — GPU sync via NVLink ✓ Fast intra-server
Each GPU computed its shard of the layer output. AllReduce sums all 8 shards so every GPU has the complete result before the next layer. Happens after every one of the 80 transformer layers — that is 80 AllReduce operations per single token generated. NVLink at 900 GB/s makes this fast within one server (<5% overhead). Across servers via InfiniBand it becomes a significant bottleneck.
5
Logit Sampling → next token selected
Final layer output (8,192-dim vector) × vocabulary matrix (8,192 × 128,256 tokens) produces a score for every possible next word. Softmax converts to probabilities. Temperature and top-p sampling selects one token. One token = one complete forward pass. A 200-word response ≈ 270 passes.
⚠ InfiniBand 400G NDR — BOTTLENECK #2 — multi-server only
Required when model exceeds one server's HBM (e.g. Llama 405B needs 7+ servers). 50 GB/s effective throughput. 80 AllReduce round-trips per output token compounds this latency.

What "70B Parameters" Actually Means

70 billion decimal numbers — each a small value like 0.00432 — stored in matrices. Not sentences, not facts. Statistical patterns encoded as numbers.

ONE Q WEIGHT MATRIX — 8192 × 8192 = 67M values
Each tiny cell = one weight value. A 70B model has ~1,000 matrices this size.

What weights encode: Not facts — statistical correlations. Weight #3,847,291 might nudge the output toward numbers when the context contains financial questions. No single weight maps to a concept. Removing weights doesn't remove facts — it reduces the model's ability to hold nuance at the edges.
⚠ MEMORY: 70B × 2 bytes (FP16) = 140 GB
H100 = 80 GB HBM. Model does not fit on one chip. Minimum 2 GPUs for weights alone. With 14K context KV cache: 3–4 GPUs for one user's request.
Quantization — Precision vs Size

Quantization reduces the decimal precision of every weight to shrink total model size. Analogy: the difference between writing "3.14159265" vs "3.14" vs "3" — same number, different precision.

It does not remove learned patterns or "delete sentences." It rounds every one of the 70 billion dials to a coarser resolution. Fine distinctions and edge-case reasoning degrade first. Core knowledge is largely preserved at INT8.

Bottlenecks + Where to Spend Money
🔴
HBM Bandwidth (within chip)
Dominant bottleneck during inference. GPU waits for weights to load from HBM between every computation. H100 (3.35 TB/s) → H200 (4.8 TB/s) = ~40% more tokens/second for the same model, same configuration.
💰 Upgrade to H200 or MI300X
🔴
HBM Capacity (model + KV cache)
Hard ceiling on model size and concurrent users. 70B FP16 + 14K context = 184 GB = 3 H100s per user. Directly limits how many users you can serve simultaneously per server.
💰 B200/MI300X (192GB) or INT4 quantization
🟡
RAG Network Latency
If vector DB is on customer infra and GPU is at provider, each RAG retrieval adds 20–200ms — often more than the GPU compute itself. This is the most overlooked bottleneck in enterprise deployments.
💰 Co-locate vector DB with GPU cluster
🟡
InfiniBand (inter-server AllReduce)
50 GB/s effective vs NVLink 900 GB/s. Compounds across 80 layers × 270 tokens per response. Keeping models on as few servers as possible (larger GPUs) reduces IB usage.
💰 Fewer servers = less IB dependency
🟡
Context Length (attention O(n²))
Doubling context from 7K to 14K tokens quadruples attention compute and KV cache memory. Long RAG pipelines and document uploads directly degrade throughput.
💰 Chunk documents smaller, use Flash Attention
🟢
Raw TFLOPS (compute power)
Counter-intuitively not the bottleneck for inference. GPU compute units are usually idle waiting for HBM. Buying more TFLOPS without more HBM bandwidth does very little.
🚫 Not worth spending on alone
🟢
NVLink (intra-server)
900 GB/s — AllReduce within a single DGX adds <5% overhead. Not a bottleneck in practice. Upgrading NVLink alone provides minimal benefit.
🚫 Not a priority
