01
The Prompt

Token Arithmetic — What Actually Enters the Machine

Before a single GPU cycle runs, your prompt is converted into tokens— integer IDs from a fixed vocabulary (~32K–128K entries). Every cost, latency, and memory calculation starts here.

Enter Your Prompt
A token is the basic unit of text that language models process — not words, not characters, but something in between. A tokenizer splits text into chunks from a fixed vocabulary (typically 32K–128K entries), where each chunk gets a numeric ID. Rules of thumb for English: 1 token ≈ 4 characters ≈ ¾ of a word. "Hello world!" → 3 tokens. "GPU" → 1 token. "antidisestablishmentarianism" → 6 tokens. Non-Latin scripts (Chinese, Arabic, emoji) use significantly more tokens per character. Why does this matter? Every API pricing model, every latency estimate, and every VRAM calculation starts with token counts.
Expected response length 256 tokens
Output / completion tokens are tokens generated by the model — the response. Each output token requires one full decode pass: reading all model weights, computing attention, producing one word-piece. Output tokens are 2–5× more expensive to produce than input tokens because they cannot be parallelised — the model must wait for token N before generating token N+1. A 256-token response is roughly 3–4 short paragraphs.
Common reference: tweet ≈ 50 tok · email ≈ 150 tok · blog post ≈ 800 tok · GPT-4 max = 4,096 tok
Model context limit 1,024 tokens
Context window is the maximum number of tokens (input + output combined) the model can "see" at once. Think of it as short-term memory. Beyond this limit, older content is forgotten. Longer context windows require more VRAM (KV cache grows linearly) and slower attention computation (O(n²) complexity). GPT-4o: 128K tokens · Claude Sonnet: 200K tokens · Llama 3.1: 128K tokens. A 200K context window on a 70B model adds ~50GB of KV cache VRAM on top of the 140GB of model weights.
Model architectural limit — independent of response length. Affects KV cache VRAM (see §2) and truncation risk. 1K ≈ short chat · 8K ≈ long doc · 32K ≈ book chapter · 128K ≈ novel · 200K = Claude max
The Token Boundary: Most tokenizers split on roughly 3–5 characters per token for English. Code, numbers, and non-Latin text use more tokens per "word." You are billed by the token—not the character.

▶ Show token cost math
How LLM pricing is calculated:
Cost = (input_tokens × $/M_input + output_tokens × $/M_output) / 1,000,000 Output tokens are typically 2-5× more expensive than input tokens — they require the full decode loop.

Why output tokens cost more:
Input (prefill) is compute-bound and can be batched efficiently → lower marginal cost.
Output (decode) is memory-bandwidth-bound and inherently sequential → higher marginal cost.

Example (GPT-4o class pricing, ~$5/$15 per 1M):
Cost = (0 × $5 + 256 × $15) / 1,000,000 = $0.0038

The Model

Parameters, Precision & Memory — What Lives in the GPU

A "70B parameter model" has 70 billion floating-point numbers. Each number takes 2–4 bytes of GPU memory depending on precision. This is the single most important constraint on which GPUs can even run the model.

Model Presets — Pick a Real Model

Select a real-world model to auto-fill the configuration below. Note: closed-source model sizes (OpenAI, Anthropic) are estimated from benchmark behaviour and industry reporting — companies do not disclose exact parameter counts.

Model Configuration
Parameter Count
What is a "parameter"? A parameter is a single number (a weight) stored in the model. During inference, every parameter is multiplied by activations to predict the next token. A 70B model has 70 billion such numbers. More parameters → better quality, but more memory and slower decode. The count is set at training time and cannot change at inference.
Examples: Llama 3 70B, Claude Sonnet-class
Numerical Precision
What is numerical precision? Every parameter (weight) is stored as a number. The precision determines how many bits are used to store each number:

FP3232-bit float — 4 bytes/param. Full precision. Used during training. Too slow and memory-hungry for inference. BF1616-bit brain float — 2 bytes/param. Half the memory of FP32 with near-identical quality. Industry standard for inference. INT88-bit integer — 1 byte/param. Requires calibration (SmoothQuant / LLM.int8). Tiny quality loss, 2× memory saving. Safe for most tasks. INT44-bit integer — 0.5 bytes/param. Requires AWQ or GPTQ quantisation. ~1–3% benchmark drop. Can make a 70B model fit on 2 GPUs instead of 4.

Analogy: Think of a painting. FP32 is the original 4K canvas. BF16 is a high-res JPEG — virtually indistinguishable. INT8 is a compressed JPEG. INT4 is a heavily compressed thumbnail — still recognisable, but artefacts appear. Precision is a physics knob: halving precision halves VRAM usage and roughly doubles how many tokens per second you can generate.
Batch Size 1
What is batch size? Instead of serving one request at a time, the GPU can process multiple requests simultaneously. Batch size = how many requests are handled in parallel. Since the model weights are already loaded into VRAM, batching amortises the weight-read cost across all requests — dramatically improving throughput (tokens/sec total) without proportionally increasing latency. A batch of 8 can often generate 6–7× more total tokens/sec than batch=1, at the cost of each user waiting slightly longer.
Larger batches amortize the model read cost across more requests simultaneously.
Context Documents None
Why does context affect memory? In enterprise use, you often inject documents into the model's context window alongside your query — this is called RAG (Retrieval-Augmented Generation) or MCP (Model Context Protocol).

These documents consume tokens in the context window. Every token in context must be stored as a key-value pair in the KV Cache — a large, fast-access memory buffer. The more documents, the more VRAM needed for the KV cache.

Real-world sizes:
• 1 document ≈ 2,000–5,000 tokens (a 4-page report or 200-line code file)
• 1 presentation ≈ 1,500–3,000 tokens (20-slide deck with speaker notes)
• 1 email thread ≈ 500–2,000 tokens (10-15 messages)

At 20 documents, you may need more VRAM than the model itself — which changes which GPU configurations are viable.
No extra context — just the current prompt and response.
What is a parameter? It's a single weight in a matrix multiplication. When generating a token, the GPU reads every parameter once—that's why larger models are slower, and why memory bandwidth (not raw FLOPS) determines decode speed.

The Critical Constraint

The Networking Penalty — Multi-GPU Communication Overhead

The point is simple: when a model spills across GPUs, communication can erase the benefit of adding more cards. NVLink keeps that tax small. PCIe can make a larger cluster slower, more expensive, or both.

The Point in One Minute
1. Why clusters happen

A 70B BF16 model is about 140 GB before KV cache and runtime overhead. Most GPUs cannot hold that alone, so the model is split across multiple GPUs.

2. Where the penalty comes from

After each layer, GPUs must synchronize partial results. That sync happens roughly 160 times per token on a 70B-class model. NVLink makes this cheap. PCIe makes it visible to the user as added latency.

3. Decision rule

First fit the model on as few GPUs as possible. Then prefer the fastest interconnect. Only after that should you compare cost per output token in the economics dashboard.

▶ Show the all-reduce latency math
All-reduce data volume per token (decode phase, hidden_dim=8192, 80 layers): bytes_per_all_reduce = 2 × hidden_dim × 2 bytes (BF16) = 2 × 8192 × 2 = 32,768 bytes = 32 KB num_all_reduces = 2 × num_layers = 2 × 80 = 160 total_comms_bytes = 160 × 32 KB = 5.12 MB per token
Bandwidth term (ring all-reduce, N GPUs): bw_time = 2 × (N-1)/N × 5.12 MB / interconnect_bandwidth NVLink 4.0 (900 GB/s, N=2): 2 × 0.5 × 5.12 MB / 900 = 0.006 ms ✓ negligible PCIe 4.0 (32 GB/s, N=4): 2 × 0.75 × 5.12 MB / 32 = 0.24 ms — growing...
Latency term (per-message base latency × number of messages): latency_time = num_all_reduces × interconnect_latency NVLink (0.8 μs, N=2): 160 × 0.8 μs = 0.128 ms PCIe (7 μs, N=8): 160 × 7 μs = 1.12 ms ← dominates
Memory read time (weight shard per GPU): read_time = (model_size_bytes / N) / gpu_bandwidth 2× H100 SXM5: 70 GB / 3350 GB/s = 20.9 ms → 47.8 tok/s 8× V100 32GB: 17.5 GB / 900 GB/s = 19.4 ms + 0.93ms comms = 20.3 ms → 49.3 tok/s

The Strategy

Legacy GPU Use Case Map — What Old Hardware Is Actually Good For

Not all workloads punish old hardware equally. The matrix below scores each GPU generation against common ML workloads. Green = viable and competitive. Red = physically impossible or economically irrational.

The strategic conclusion: Legacy GPUs (V100, A40) and mid-tier options are not worthless— they're mismatched to modern frontier workloads. The question for GPU lifecycle strategy is: which workloads map to their constraints (limited VRAM, lower bandwidth, PCIe-only networking)?
Summary: Where Legacy GPUs Win (and Lose)
✓ Legacy / Mid-tier GPUs Still Make Sense
  • 🟢 Text/image embedding generation (RAG)
  • 🟢 Batch classification (latency-tolerant)
  • 🟢 LoRA fine-tuning on 7B–13B models
  • 🟢 Stable Diffusion / image generation
  • 🟢 Small model inference (<13B params)
  • 🟢 A40 (48GB): single-GPU 34B inference
  • 🟢 A30 (NVLink): efficient 2-GPU 13B setups
  • 🟢 Dev/test/experimentation workloads
⚠ Legacy GPUs: Use With Caution
  • 🟡 13B–34B inference (quantized only)
  • 🟡 Medium-context batch jobs (<8K tokens)
  • 🟡 Offline document summarization
  • 🟡 Price-sensitive, latency-tolerant APIs
  • 🟡 Multi-LoRA serving (limited concurrency)
✗ Legacy GPUs Cannot Compete
  • 🔴 70B+ real-time inference
  • 🔴 Long-context (>16K tokens)
  • 🔴 Pre-training / full fine-tuning at scale
  • 🔴 Speculative decoding (needs fast interconnect)
  • 🔴 Multi-modal frontier models
  • 🔴 Sub-100ms TTFT SLA requirements
Methodology & Data Sources
GPU specs sourced from NVIDIA product pages and third-party benchmarks. Spot prices are representative estimates from CoreWeave, Lambda Labs, and AWS (H100/A100, 2024–2025) and change frequently. Inference calculations use the roofline model: decode is modeled as memory-bandwidth-bound (reading all weights per token); prefill is modeled as compute-bound. All-reduce latency uses ring-allreduce formulation. Actual performance varies with batch scheduling, KV cache management (vLLM PagedAttention, etc.), quantization kernels (AWQ/GPTQ), and driver/firmware versions.

This tool is for strategic evaluation and education — not a substitute for empirical benchmarking. Run vllm bench or llm-perf against your specific workload.
